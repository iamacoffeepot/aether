# ADR-0107: Immediate-mode UI

- **Status:** Proposed
- **Date:** 2026-06-13

Amends **ADR-0105** (the text + textured-quad render surface) by adding the one render primitive a UI needs that ADR-0105 left out. References **ADR-0066** (the render/camera overlay pass UI draws on), **ADR-0021 / ADR-0068** (the input streams a click rides), and **ADR-0099** (the typed `ctx.actor::<Cap>()` addressing a guest UI sends through).

## Context

UI is hand-composed today, and the one filled rectangle the engine draws — the locomotion HUD's health bar — is a camera-ray-projected world rectangle assembled from two `DrawTriangle`s in `aether-kit`'s `Hud` projector (`crates/aether-kit/src/runtime.rs`), precisely because the render surface has no screen-space filled-rect primitive. The HUD pins those triangles to a fixed depth along the camera ray so they hold their screen position as the camera orbits — real work to stand in for a rectangle the surface cannot draw directly.

The pieces a UI wants are otherwise all shipped. `aether.text.draw` lays out glyphs and `aether.render.draw_textured_quads` draws alpha-blended images, both on the same screen-space overlay pass (ADR-0105). The key and mouse streams (ADR-0021 / ADR-0068) deliver the cursor (`MouseMove`), the window size (`WindowSize`), and the click edge (`MouseButton`). Nothing composes those into panels, labels, and buttons a component lays out each frame, and text became load-bearing for exactly this surface once HUD and world labels landed (#1715).

Two consumers drive that composition, and they sit differently against a frame loop — the distinction the design turns on. A component author runs inside a tick loop and re-sends its draw every frame. An LLM driving over MCP `send_mail` is not in a frame loop; it pokes one-shot mail and reads a reply. Interactive immediate-mode UI fits the author directly: the actor that draws a button already holds, frame to frame, the state a click resolves against — the cached cursor, the window size, the press edge. It fits the LLM through a running component the LLM controls — a static panel emitted as one-shot mail renders fine, a live button polled per frame over the wire does not. The design serves the author's interactive case and the LLM's static case, and a per-frame button-mail round trip serves neither.

## Decision

UI is guest-side immediate-mode layout in `aether-kit`, drawn on a render surface that gains exactly one new primitive. Three parts.

**1. Layout and hit-testing are pure functions in `aether-kit`, not a chassis capability.** A `ui` module exposes `panel`, `label`, `bar`, and `button` as functions the drawing component calls each frame; they emit `draw_solid_quads` / `aether.text.draw` mail and report which controls the cursor is over. The click never leaves the actor that drew the button, because that actor already holds the state a UI needs — the cached cursor from `MouseMove`, the window size from `WindowSize`, the press edge from `MouseButton` (`crates/aether-kinds/src/lib.rs`). This is the zero-round-trip pattern the kit already uses for ground-plane picking: `pick_world` in `crates/aether-kit/src/runtime.rs` casts a ray from the cached cursor through the camera onto the ground without a single host round trip, and a button's hit test is the same shape against a screen rect. Resolving the click inside the actor that owns the rect is what keeps UI free of retained cross-seam state.

**2. The render surface gains one primitive it genuinely lacks: a flat-colored quad.** `aether.render.draw_solid_quads { space: QuadSpace, quads: Vec<SolidQuad> }` with `SolidQuad { x, y, width, height, color: [f32; 4] }` draws a batch of flat-colored, alpha-blended rectangles. It reuses the existing `QuadSpace` discriminant (`Screen` / `World`, `crates/aether-kinds/src/lib.rs`) and rides the textured-quad overlay pass: the cap tints a lazily-created internal 1×1 white texture by each quad's `color`, so the batch flows through the same blended overlay pipeline `draw_textured_quads` already records — no new pipeline, no new pass. `color` is an explicit linear-RGBA array mirroring `TexturedQuad.tint`, and the kind's arity mirrors `DrawTexturedQuads` minus the `texture_id` the white-texture trick supplies internally. `RenderCapability::on_draw_solid_quads` accumulates the batch into the per-frame quad accumulator exactly as `on_draw_textured_quads` does (`crates/aether-capabilities/src/render.rs`); `HeadlessRenderCapability::on_draw_solid_quads` is a no-op, mirroring its `on_draw_textured_quads` so desktop-built components run clean on headless.

**3. Everything resends per frame.** The `ui` functions emit their quad and text mail every frame the UI should appear, the same immediate-mode contract `draw_triangle`, `draw_textured_quads`, and `aether.text.draw` already carry. There is no retained UI object, no create/update/destroy, and no UI state living at any mailbox.

## Consequences

- **Positive** — no retained UI state lives at any seam. The drawing actor owns its layout and its hit state as plain fields, the render cap owns only a per-frame accumulator it already clears each frame, and no mailbox holds a `(button → rect → owner)` table that has to survive across frames. UI composes entirely on surfaces that already shipped — `draw_solid_quads` is the lone addition, and it is one kind plus two ten-line handlers.
- **Positive** — an LLM can emit a static UI today with no component at all: a `draw_solid_quads` batch for the panel and a `aether.text.draw` for the label, sent as one-shot mail each render it wants them visible. The interactive path is the same vocabulary wrapped in a component the LLM loads and controls.
- **Neutral / cost** — `aether-kit` gains a `ui` module, and the kind vocabulary gains one kind (`DrawSolidQuads` plus its `SolidQuad` element). The render cap gains one accumulate handler and one headless no-op, both copies of the textured-quad pair.
- **Negative / risk** — `aether.mouse_button` carries no payload (`pub struct MouseButton;`, a unit kind in `crates/aether-kinds/src/lib.rs`): it signals a press with no button discriminant and no release. v1 buttons are therefore left-click-activate-only — no right-click, no drag, no press-and-hold. World-space UI panels wait on the `QuadSpace::World` anchor path landing; `World` ships in the vocabulary but warn-drops at encode until that path is implemented (ADR-0105), so v1 UI is `Screen`-space only.
- **Follow-on** — a `MouseButton { button, pressed }` payload kind unlocks right-click, release, and drag, and it is its own change with its own input-stream and chassis-publish surface. A dedicated `aether-ui` crate is worth extracting once a second non-kit consumer earns it; until then the `ui` module lives in `aether-kit` where the only consumer is.

## Alternatives considered

- **An `aether.ui` chassis capability that emits click mail** — rejected: to turn a press into a "button N clicked" message the cap must retain a `(button → rect → owner)` table and rebuild it every frame as the layout shifts, which is retained UI state wearing immediate-mode clothing. The actor that drew the button already holds that mapping for free; routing the click out and back manufactures the seam this design exists to avoid.
- **A white-texture SDK helper instead of a new kind** — rejected: handing callers a `create_texture` of a 1×1 white pixel and a `draw_textured_quads` wrapper leaks a session-scoped `texture_id` an MCP caller has to thread through, and it is less self-describing than a `draw_solid_quads` kind that mirrors `draw_textured_quads` and names `color` outright. A flat-colored quad is a primitive the surface should expose as a kind, not a recipe each caller reassembles.
- **Retained UI objects (create / update / destroy)** — rejected: ADR-0105 already declined retained text for diverging from the immediate-mode render contract, and the same reason holds here. Per-frame resend keeps one redraw model across triangles, textured quads, text, and now solid quads; a retained layer can come later behind the same kinds if profiling ever demands it.
