# ADR-0107: Immediate-mode UI

- **Status:** Proposed
- **Date:** 2026-06-13

Amends **ADR-0105** (the text + textured-quad render surface): adds the one flat-fill render primitive a UI needs and a native UI capability that composes the surface — symmetric with `aether.text`, ADR-0105's own composing capability. References **ADR-0066** (the render/camera overlay pass UI draws on), **ADR-0021 / ADR-0068** (the input streams a click rides), and **ADR-0099** (the typed `ctx.actor::<Cap>()` addressing a component reaches the UI cap through).

## Context

UI is hand-composed today, and the one filled rectangle the engine draws — the locomotion HUD's health bar — is a camera-ray-projected world rectangle assembled from two `DrawTriangle`s in `aether-kit`'s `Hud` projector (`crates/aether-kit/src/runtime.rs`), precisely because the render surface has no screen-space filled-rect primitive. The HUD pins those triangles to a fixed depth along the camera ray so they hold their screen position as the camera orbits — real work to stand in for a rectangle the surface cannot draw directly.

The pieces a UI wants are otherwise all shipped. `aether.text.draw` lays out glyphs and `aether.render.draw_textured_quads` draws alpha-blended images, both on the same screen-space overlay pass (ADR-0105). The key and mouse streams (ADR-0021 / ADR-0068) deliver the cursor (`MouseMove`), the window size (`WindowSize`), and the click edge (`MouseButton`). Nothing composes those into panels, labels, and buttons a component lays out each frame, and text became load-bearing for exactly this surface once HUD and world labels landed (#1715).

Two consumers drive that composition, and they sit differently against a frame loop. A component author runs inside a tick loop and re-sends its draw every frame. An LLM driving over MCP `send_mail` is not in a frame loop; it pokes one-shot mail and reads a reply. The question the design turns on is where the composition lives — folded into each consumer, or housed in one capability both mail. A capability serves both from a single surface: the author mails its widget set each tick, and the LLM mails the same widgets directly with no frame loop to host, reading a button's activation back as a reply. A guest-side helper module is the same layout and hit-test compiled afresh into every wasm consumer, and out of reach of the LLM except through a component it must run.

## Decision

UI is a native capability, `aether.ui`, that composes `aether.text`, the textured-quad surface, and one new flat-fill render primitive into panel, label, bar, and button widgets. It lives in `aether-capabilities` next to `aether.text` and is symmetric with it: `aether.text` is a native CPU-only cap that composes the quad surface into glyphs, and `aether.ui` is a native cap that composes both into widgets.

Native is the load-bearing choice for three reasons. It matches the engine's composition model — a component mails a capability that owns the composition, exactly as it mails `aether.text` to compose glyphs and `aether.render` to compose quads. It yields one reusable UI surface every component shares, native and wasm alike, instead of layout and hit-testing recompiled into each guest. And the retained-state objection that would normally argue against housing UI in a capability dissolves under immediate mode: the cap holds only the current frame's widget rects, rebuilt each frame, so it persists no UI state across frames.

Four parts.

**1. `aether.ui` is a native capability in `aether-capabilities`.** It mirrors `TextCapability` (`crates/aether-capabilities/src/text.rs`): a `#[actor] impl NativeActor`, `NAMESPACE = "aether.ui"`, no GPU access — it only sends mail. A component addresses it by type (`ctx.actor::<UiCapability>()`, ADR-0099) and mails its widget set each frame; the cap composes that set the same tick onto `aether.render` and `aether.text`, the two surfaces it reaches through `ctx.actor::<RenderCapability>()` / `ctx.actor::<TextCapability>()`. A dedicated `aether-ui` crate is the documented extraction once a second consumer earns it; until then the cap lives beside `aether.text`.

**2. Immediate-mode: the component re-sends its widget set each frame.** Panel, label, bar, and button widgets arrive as UI draw mail (`aether.ui.{panel,label,bar,button}`) to the `aether.ui` mailbox; the cap composes each into `draw_solid_quads` fills (panel, bar, button background) and `aether.text.draw` strings (label and button text, naming the `font_id` the component loaded through `aether.text.load_font`) on the same tick — the immediate-mode contract `draw_triangle`, `draw_textured_quads`, and `aether.text.draw` already carry. The cap holds only the current frame's widget rects — rebuilt each frame like the render cap's per-frame quad batch — so it carries no retained UI state, no create/update/destroy, and no UI object living at the mailbox.

**3. Interaction: the cap hit-tests clicks and reports activation by mail.** The cap subscribes to the mouse streams from its `wire` hook (`MouseMove` for the cursor, `MouseButton` for the click edge; ADR-0021 / ADR-0068). On a click it hit-tests the cursor against the current frame's button rects and, on a hit, mails `aether.ui.clicked { id }` back to the source of that button's draw mail — the owning component, identified by the widget mail's sender, which the cap records alongside each button rect. The `id` is the value the component assigned the button; the activation echoes it, so a component routing several buttons tells them apart with no per-button mailbox. Resolution is one frame latent: the click tests against the rects the last frame's draw mail left.

**4. The render surface gains `draw_solid_quads`.** `aether.render.draw_solid_quads { space: QuadSpace, quads: Vec<SolidQuad> }` with `SolidQuad { x, y, width, height, color: [f32; 4] }` draws a batch of flat-colored, alpha-blended rectangles. It reuses the existing `QuadSpace` discriminant (`crates/aether-kinds/src/lib.rs`) and rides the textured-quad overlay pass — the render path tints a lazily-created internal 1×1 white texture by each quad's `color`, so the batch flows through the same blended pipeline `draw_textured_quads` already records, with no new pipeline and no new pass. `color` is an explicit linear-RGBA array mirroring `TexturedQuad.tint`, and the kind's arity mirrors `DrawTexturedQuads` minus the `texture_id` the white-texture trick supplies internally. `RenderCapability::on_draw_solid_quads` accumulates into the per-frame quad accumulator exactly as `on_draw_textured_quads` does (`crates/aether-capabilities/src/render.rs`); `HeadlessRenderCapability::on_draw_solid_quads` is a no-op, mirroring its `on_draw_textured_quads` so desktop-built components run clean on headless.

## Consequences

- **Positive** — one UI surface serves every consumer. A component mails its widgets and the cap composes them; the same widget mail drives the cap from an LLM over MCP `send_mail` directly, with no frame loop to host in a guest — a panel-plus-label batch sent each render it wants visible, a button whose `aether.ui.clicked` reply the LLM reads back. The cap is symmetric with `aether.text` and `aether.render`: a CPU-only native actor that composes the surface below it, the shape ADR-0105 already established for text.
- **Neutral / cost** — a new capability joins `aether-capabilities`, plus the widget draw kinds (`aether.ui.{panel,label,bar,button}`), the `aether.ui.clicked` activation kind, and the `draw_solid_quads` render kind with its `SolidQuad` element. The cap subscribes to the mouse streams — the cursor and click edge the hit test reads — and holds the current frame's button rects to test against; the rects rebuild each frame, so no UI layout persists across frames.
- **Negative / risk** — `aether.mouse_button` carries no payload (`pub struct MouseButton;`, a unit kind in `crates/aether-kinds/src/lib.rs`): it signals a press with no button discriminant and no release. v1 buttons are therefore left-click-activate-only — no right-click, no drag, no press-and-hold. World-space UI waits on the `QuadSpace::World` anchor path; v1 widgets are `Screen`-space only.
- **Follow-on** — a `MouseButton { button, pressed }` payload kind unlocks right-click, release, and drag, and it is its own change with its own input-stream and chassis-publish surface. A dedicated `aether-ui` crate is worth extracting once a second consumer earns it. Richer widgets — sliders, text fields, scrolling panels — extend the same draw-mail vocabulary behind the same cap.

## Alternatives considered

- **Guest-side SDK layout helpers in `aether-kit`** — `panel` / `label` / `bar` / `button` as pure functions a drawing component calls each frame, emitting quad and text mail and hit-testing the cached cursor in-process (the zero-round-trip shape `pick_world` already uses in `crates/aether-kit/src/runtime.rs`). Rejected: it recompiles the UI surface — layout and hit-testing — into every wasm guest that wants UI and splits one surface across its consumers, and it reaches an LLM only through a component the LLM must run. The engine's model is for a component to mail a capability that owns the composition, the way it already mails `aether.text` and `aether.render`; housing UI the same way is the maintainer's explicit call.
- **A white-texture SDK helper instead of a `draw_solid_quads` kind** — hand callers a `create_texture` of a 1×1 white pixel and a `draw_textured_quads` wrapper. Rejected: it leaks a session-scoped `texture_id` an MCP caller has to thread through, and it is less self-describing than a `draw_solid_quads` kind that mirrors `draw_textured_quads` and names `color` outright. A flat-colored quad is a primitive the surface should expose as a kind, not a recipe each caller reassembles.
- **Retained UI objects (create / update / destroy)** — rejected: ADR-0105 already declined retained text for diverging from the immediate-mode render contract, and the same reason holds here. Per-frame resend keeps one redraw model across triangles, textured quads, text, solid quads, and now widgets; a retained layer can come later behind the same kinds if profiling ever demands it.
