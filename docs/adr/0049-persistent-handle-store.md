# ADR-0049: Persistent handle store across substrate restart

- **Status:** Proposed
- **Date:** 2026-04-25

## Context

ADR-0045 shipped the substrate-global handle store as an in-memory `HashMap<HandleId, HandleEntry>` with refcounting, LRU eviction, and a pinned flag. ADR-0048 added content-addressed transform handle ids: a transform applied to the same inputs always produces the same handle id, so two callers wiring identical compute share one cached output. These two together cover within-process pipeline composition cleanly.

They don't cover **across-process** continuity. A substrate restart drops the entire handle store. A content-gen pipeline (ADR-0046) generates an image, the image's bytes live in the store under a content-addressed handle id, the substrate exits, and the next run starts from scratch — even though the handle id is deterministic and the bytes are recomputable. The expensive recomputation (image generation: $0.05-0.30 per call, 10-100× slower than text) happens twice for nothing.

This is the forcing function ADR-0046 §5 names. The image-as-input cascade — a scene render's reference list includes object-reference sheets generated by earlier renders — only works if handles published yesterday are addressable today. Without persistence, every restart breaks the cascade and every recipe re-author from scratch.

ADR-0045 §10 parks persistence to "Phase 4+ (separate ADRs)" with the wire-format guarantee that adding it requires no protocol changes. This ADR is that separate ADR. Scope: on-disk layout, restore semantics, eviction-across-restart, schema-evolution handling, chassis coverage.

Out of scope: distributed handle stores (multiple substrates sharing one persistent store) and DAG state persistence (a DAG that started before restart resuming after — see ADR-0047 §7). Both are forward-compatible additions; this ADR commits to the storage primitive and the in-process recovery path only.

## Decision

### 1. What persists, what doesn't

Persistent handles are a strict subset of all handles:

- **Persistent: content-addressed transform handles** (ADR-0048 §4). The handle id is deterministic from `(component_mailbox, transform_index, input_handle_ids)`; restoring it by id and recomputing it from scratch produce the same id. Persistence is meaningful.
- **Persistent: pinned handles** of any origin. Asset pipelines that explicitly pin a handle (`aether.handle.pin` per ADR-0045 §9) declare the intent that it should outlive the current process. Pinned handles persist even if their origin is non-deterministic — the substrate trusts the caller's pin as an explicit "save this" directive. Re-creating a pinned source handle from scratch isn't possible (the source observation is gone), but the bytes are still there for any caller that holds the pinned id (e.g., as part of a recipe).
- **Not persistent: source handles** (ADR-0045 §3). Source ids are ephemeral monotonic; a Fetch today and a Fetch tomorrow are two distinct observations and get two distinct ids. Persisting source bytes by id would let a caller "restore" a handle that the next observation won't reproduce. Sources stay in-memory only, evicted at process exit. The exception above (pinned source handles) covers the "I want to capture this fetch as a recipe input" case.
- **Not persistent: DAG state.** ADR-0047 §7 keeps DAG state in-memory only. A restarted substrate doesn't auto-resume DAGs; the harness or recipe layer re-submits. Re-submitted DAGs find their content-addressed transform-output handles already on disk (this ADR's contribution) and skip recomputation; sources re-dispatch.

This split keeps the persistence semantics honest: only handles whose identity is reproducible across restart get the on-disk slot.

### 2. On-disk layout

The substrate owns a content-addressed file tree under a per-substrate data directory. (The same `<hash[0:2]>/<hash[2:]>.<ext>` directory shape was practically validated by ADR-0046's Spike B binary cache, where it carried 600-700 KB PNGs across multiple cascade-spanning runs cleanly. The 256-bucket prefix-shard granularity is the right size for tens of thousands of entries; rebalancing isn't needed in practice until much larger scales.)

```
${AETHER_HANDLE_STORE_DIR}/
  v1/
    entries/
      <hash[0:2]>/
        <hash[2:]>.bin    # canonical-bytes encoding of the kind value
        <hash[2:]>.meta   # postcard-encoded HandleMeta sidecar
    index.bin             # optional summary index, rebuildable from entries/
    pinned.set            # set of HandleIds the user has pinned
```

Default `${AETHER_HANDLE_STORE_DIR}` resolves to `dirs::data_dir()/aether/handles` (e.g., `~/Library/Application Support/aether/handles` on macOS, `~/.local/share/aether/handles` on Linux). Override via the `AETHER_HANDLE_STORE_DIR` env var.

Per-handle artifacts are two files:

- **`<hash>.bin`** — the canonical-bytes encoding of the kind value (ADR-0032). Restoring this file's contents back into the in-memory `HandleEntry.bytes` field is a copy-from-disk; the bytes don't need re-decoding because the on-disk and in-memory encodings are identical.
- **`<hash>.meta`** — a `HandleMeta` postcard struct:

```rust
struct HandleMeta {
    schema_version: u8,        // 1
    handle_id: u64,            // for collision detection on restore
    kind_id: u64,              // schema-hash of the kind type (ADR-0030)
    transform_origin: Option<TransformOrigin>,  // None for pinned source handles
    bytes_len: u32,            // for cheap consistency check vs .bin file
    created_at: u64,           // millis since unix epoch (informational)
    pinned: bool,
}

struct TransformOrigin {
    component_mailbox: u64,    // who computed this
    transform_index: u32,
    input_handle_ids: Vec<u64>, // inputs that produced this output, slot-ordered
}
```

The `transform_origin` field is what makes persistence interpretable: a handle file on disk knows what transform produced it and which inputs went in. This is the substrate's local record of the same provenance ADR-0046 §7 lifts to the application layer.

`pinned.set` is a small file (one `u64` per pinned handle) the substrate writes whenever pin/unpin mail arrives. Cheap to serialise; load eagerly at startup.

`index.bin` is an optional summary the substrate writes on graceful shutdown — a `HashMap<HandleId, IndexEntry>` covering everything in `entries/`. Cheap to load; if missing or corrupt, the substrate falls back to scanning `entries/` directory tree at boot to rebuild it. The index is a startup latency optimization, not a correctness primitive.

### 3. Persist-on-resolve, restore-on-demand

**Write path.** When a transform completes (ADR-0048 §3, step 5), the substrate stores the output bytes in the in-memory handle store keyed on the content-addressed handle id, then writes `<hash>.bin` and `<hash>.meta` to disk via the same atomic tmp+rename pattern as ADR-0041's `LocalFileAdapter`. Write failure (disk full, permission denied) logs a warning to engine_logs but doesn't fail the transform — the in-memory entry is still valid for this run. Persistence is best-effort within the run; the caller doesn't see a different return value.

The disk write happens on the substrate's IO thread, off the actor-per-component dispatch path (ADR-0038). Transforms don't block on disk persistence.

**Read path.** When a DAG executor sees a `Transform` node and computes its content-addressed handle id (per ADR-0048 §4), it first checks the in-memory store. Miss → check the on-disk store: if `<hash>.bin` + `<hash>.meta` exist and `meta.handle_id` matches, load the bytes into the in-memory store with refcount = 1 and return the entry. Miss → dispatch the transform via wasmtime as usual.

The on-disk check is one or two filesystem stat + open syscalls per cache miss. For pipelines with high-cardinality transforms (thousands of unique input combinations) this is cheap. For pipelines that thrash (lots of distinct DAGs, lots of distinct transform invocations) the substrate populates an in-memory negative-cache (`HashSet<HandleId>` of "checked, not on disk") to avoid re-statting on repeated misses; bounded at 8K entries with FIFO eviction.

**Restore semantics.** A substrate restart does *not* eagerly load all handles into memory. The on-disk tree is the truth; the in-memory store is a hot cache on top. At boot:

1. Read `pinned.set` and pin those handle ids (the in-memory entries materialize lazily on first access).
2. Read `index.bin` if present; populate a sparse in-memory index of `HandleId → file path` for fast lookups. If absent, walk `entries/` and rebuild.
3. Disk-resident handles are not loaded into memory; they materialize on access, identical to a cache miss.

This means a substrate with 50GB of cached image bytes on disk doesn't hold 50GB in memory at boot. The in-memory store stays bounded by `AETHER_HANDLE_STORE_MAX_BYTES` (default 256MB per ADR-0045); on-disk store grows until `AETHER_HANDLE_STORE_DISK_BUDGET_BYTES` (default 16GB, configurable) triggers eviction (§5).

### 4. Pinned and refcount semantics across restart

The persistence model needs to reconcile two lifecycle dimensions: **refcount** (in-process, transient) and **pinned** (durable, user-declared).

- **Refcount.** A handle's refcount reflects "who in this process is holding it." Refcount-zero entries are eligible for in-memory eviction (LRU per ADR-0045 §9). On-disk persistence is *orthogonal*: a refcount-zero entry can still be on disk (because pinned, or because it was just written and not yet evicted). At restart, every restored entry starts with refcount = 0 — no in-process consumers exist yet.
- **Pinned.** Pinned handles are durable: pin status survives restart via `pinned.set`. The pin is an explicit user directive that says "keep this handle available; don't evict regardless of refcount or LRU pressure." Pinned + refcount-0 handles persist on disk and don't materialize in memory until accessed.
- **Unpinning** removes the handle from `pinned.set` and (if refcount = 0) makes the on-disk artifacts eligible for disk eviction (§5). The in-memory entry follows normal LRU after unpin.
- **Explicit deletion** is not a primitive in v1. There's no `aether.handle.purge` mail — eviction handles unwanted entries via the pressure mechanisms below. A future ADR can add explicit deletion if a workflow forces it (e.g., GDPR-style "remove this image").

### 5. Disk eviction

When `total_disk_bytes > AETHER_HANDLE_STORE_DISK_BUDGET_BYTES`, the substrate evicts:

1. **Eviction candidates** are entries that are *both* refcount-0 in-memory *and* unpinned. Anything pinned or in-use stays.
2. **Eviction order** is LRU by `meta.created_at` for v1 (a true last-access timestamp would require updating `.meta` on every read, which is unacceptable disk write amplification). created_at-LRU is a heuristic that works well enough for content-gen workflows where older content tends to be either promoted-and-untouched or stale.
3. **Eviction is two-phase**: delete `<hash>.bin` first (frees the bytes), then delete `<hash>.meta` (drops the index entry). A crash between phases leaves a meta-orphan that a startup scrub catches; the scrub deletes orphans on boot.
4. **Eviction runs on a slow tick** (default 60s) when over budget; bursts of writes don't synchronously trigger eviction. The disk-bytes ledger is approximate (updated on each write) — a brief over-budget excursion is fine.

A future ADR can introduce a richer eviction policy (e.g., "evict transform handles before pinned source handles" or "preserve handles with high in-degree for downstream consumers") if usage patterns demand it.

### 6. Schema evolution

A handle's bytes encode a kind value at the schema version current at write time. If a transform's output kind schema changes (a new field added to `ComposedPrompt`, an enum variant renamed), the on-disk bytes from before the change are now decoding against an older schema.

Two protections:

1. **`meta.kind_id` is the schema-hash of the kind type at write time** (ADR-0030: `kind_id = fnv1a_64(KIND_DOMAIN ++ canonical(name, schema))`). On restore, the substrate compares the on-disk `meta.kind_id` to the current kind registry's `kind_id` for the same kind name. Match → bytes are valid, decode normally. Mismatch → the on-disk entry is stale; the substrate deletes it (`<hash>.bin` + `<hash>.meta`) and treats it as a cache miss. The transform re-runs and writes a fresh entry under the new schema-hash.
2. **Schema-version-mismatch on `meta.schema_version`** (the field is set to 1 in v1, increments if `HandleMeta` itself evolves). Mismatch on this field is a substrate-version skew, not a kind-schema skew; the substrate logs and treats as miss.

This means schema evolution is automatic: change a kind's schema, all on-disk handles for that kind invalidate naturally on next access. No migration tool required. The cost is that schema-evolution-during-active-iteration discards potentially useful cached work; the iteration workflow can pin the handles it wants to preserve.

### 7. Concurrency

The substrate owns its `${AETHER_HANDLE_STORE_DIR}` exclusively. Two substrates pointing at the same directory is undefined behaviour — atomic tmp+rename writes don't synchronize across processes, and the in-memory eviction state would diverge.

The substrate writes a `lock.pid` file at startup with its PID; if the file exists and the PID is live, the substrate exits with a clear error pointing at `AETHER_HANDLE_STORE_DIR` and the holding PID. This is the same pattern as `cargo`'s lockfile and is enough for the v1 single-substrate-per-store assumption.

Multi-substrate stores (sync between machines, distributed cache) are deferred. ADR-0046 §6 names the three-layer storage shape (authored / cache / promoted) that already covers cross-machine work via the promoted layer; the cache is intentionally per-machine.

### 8. Migration and backward compatibility

The store didn't exist before v1, so there's no migration. A substrate that boots into a directory without a `v1/` subdirectory creates one and starts fresh. A future v2 of this layout would either:

- Add a `v2/` subdirectory beside `v1/` and migrate lazily on access (read from v2 first, fall back to v1, write to v2 on transform completion); periodic background migration or eventual GC of `v1/`.
- Or hard-deprecate `v1/` with a one-time conversion at boot.

The `v1/` namespacing is the forward-compatibility hook; the choice of migration strategy is for the v2 ADR.

### 9. Chassis coverage

- **Desktop** — full persistence. `dirs::data_dir()/aether/handles` default. Persistence is the headline benefit.
- **Headless** — full persistence. Identical semantics to desktop. Headless content-gen workloads benefit most: long-running pipelines that span days of authoring.
- **Hub** — no persistence. The hub doesn't host handles per ADR-0045 §11; nothing to persist. `AETHER_HANDLE_STORE_DIR` is ignored.

Per-chassis disk budget overrides via `AETHER_HANDLE_STORE_DISK_BUDGET_BYTES` work uniformly across desktop and headless.

### 10. Observability

The hub MCP surface gains one tool:

- **`mcp__aether-hub__describe_handles(engine_id, max?)`** — returns a summary of the substrate's persistent store: total entry count, on-disk byte budget vs in-use, pinned-handle count, top-N entries by size, top-N entries by recency. Useful for triaging "why is my handle store at the disk-budget cap" without ssh-ing into the substrate's machine.

ADR-0023's `engine_logs` carries write-failure diagnostics, eviction events, and schema-mismatch invalidations.

A separate `mcp__aether-hub__pin_handle(engine_id, handle_id)` and `mcp__aether-hub__unpin_handle(engine_id, handle_id)` would be useful but are a thin wrapper around the existing `aether.handle.pin` / `aether.handle.unpin` mail kinds (ADR-0045 §9); deferred until a workflow forces it.

## Consequences

### Positive

- **Image-as-input cascade works.** ADR-0046 §5's persistent computation graph — yesterday's image-gen output is today's render input — becomes addressable. The headline forcing function lands.
- **Restart latency stays low.** Lazy materialization means a substrate with a large persistent store boots in roughly the same time as a fresh one; only the index loads eagerly. No "wait while we warm the cache" startup.
- **Cost of recomputation drops to zero for cached transforms.** A pipeline whose recipe pins specific handles can re-run after restart and pay only for cache misses. ADR-0046's cost-asymmetric regen story (auto-cascade text, queue images for review) becomes much cheaper on the auto-cascade side because the substrate-level cache complements the application-level recipe-driven cache.
- **Schema evolution is automatic.** Kind schema changes invalidate stale on-disk entries via `kind_id` mismatch; no migration tooling required.
- **No new wire format.** Persistence is purely a substrate-internal mechanism; ADR-0045's `Ref<K>` wire is unchanged. Components don't know whether a handle they're consuming is in-memory or just-restored from disk.

### Negative

- **Disk budget is a new operational concern.** A substrate left running for weeks accumulates handles; `AETHER_HANDLE_STORE_DISK_BUDGET_BYTES` needs to be sized for the workload. Default 16GB is enough for thousands of cached text blocks and a few hundred reference images. Spike B's image-gen runs at ~600-700 KB per PNG put empirical numbers on the budget: a 16GB cap holds roughly 25,000 image-tier handles, more than enough for active iteration but light enough that a heavy multi-day image-gen workload could plausibly tune up. The defaults are starting points, not ceilings.
- **created_at-LRU is a heuristic.** True last-access tracking would require per-read disk writes (unacceptable amplification) or in-memory tracking that's lost on restart. created_at is good enough for v1 but means a handle that's been on disk a long time gets evicted before a recently-written one even if the long-resident one is more frequently accessed. A future ADR can add bucket-counter access tracking if eviction quality matters.
- **Single-substrate-per-store lockfile is brittle.** A substrate that crashes leaves a stale `lock.pid` whose PID may have been reused by another process. Mitigation: the lockfile check tests `kill -0 <pid>` which works for live-process detection on Unix, but a crash-during-restart race could prevent restart. v1 acceptable; future ADR can add a more robust locking primitive.
- **No explicit purge.** A user who wants to remove a specific handle from disk has to evict it via memory pressure (refcount-0 + unpinned + LRU) rather than directly. For sensitive content (a render the user wants to discard), this is awkward. Future ADR adds `aether.handle.purge` if a workflow forces it.
- **Schema-evolution-during-iteration loses cached work.** A developer iterating on a kind schema regenerates everything that uses it. Pinning preserves what the user wants to keep; the rest invalidates. Acceptable trade for automatic correctness.

### Neutral

- **In-memory store semantics unchanged.** ADR-0045 §2's refcount + LRU + pinned + max-bytes apply uniformly. The in-memory store doesn't know or care whether a given entry's bytes are also on disk.
- **`Ref<K>` wire unchanged.** Persistence is a substrate-internal mechanism. Senders and consumers of `Ref::Handle { id, kind_id }` don't know whether the handle resolves from memory, disk, or recompute.
- **DAG persistence is separate concern.** ADR-0047 §7 keeps DAG state in-memory; this ADR persists handles only. A re-submitted DAG with the same descriptor benefits from on-disk transform handles but has to re-validate and re-dispatch from scratch.

## Alternatives considered

- **Persist everything, including source handles.** Source ids are ephemeral monotonic per ADR-0045 §3; persisting them by id would let a caller "restore" a handle that the next observation produces under a different id. Rejected; persistence is meaningful only for content-addressed and pinned handles.
- **SQLite for the store backend.** Single-file storage, atomic transactions, fast queries by id. Rejected for the same reasons ADR-0046 §6 rejected SQLite for the application-level cache: file-tree storage gives visible diffs (for inspection), atomic-enough writes (content-addressed = write-once), and degrades gracefully on partial failures (a corrupt file is one entry lost, not the whole DB). A SQLite index on top of the file tree is a forward-compatible addition if reverse walks become slow.
- **Eager restore at boot (load everything into memory).** Startup latency becomes proportional to store size — a 50GB store takes minutes to boot. Rejected; lazy materialization is the right default. A `--prewarm` flag could opt into eager loading for short-substrate-lifetime workflows; deferred to follow-up.
- **Per-handle-kind directories instead of `entries/<hash[0:2]>/<hash[2:]>`.** Easier to find "all `Image` handles" but breaks content-addressing locality (two callers wiring the same `parse_json` write to the same shard naturally with the hash-prefix scheme). Rejected; the kind-id is in `meta` for kind-filtered queries, and `describe_handles` MCP can index by kind on demand.
- **Sync-write persistence (block transform completion until disk write completes).** Stronger durability — a transform that completes is on disk before the dispatch returns. Rejected for v1 because transform completion is on the substrate's IO-not actor-thread, and synchronously waiting blocks the executor's parked-mail flush. The async write loses durability guarantees in the case of substrate crash between transform completion and disk write — acceptable because the next run recomputes the same handle id and writes again.
- **Multi-substrate distributed store (Redis, S3, etc.).** Useful for sharing cache across machines. Rejected for v1 — the application-level promoted layer (ADR-0046 §6) covers cross-machine sharing; substrate-level caches stay per-machine. Distributed stores are a future ADR if a forcing function emerges.
