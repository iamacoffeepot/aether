# ADR-0095: Binding-agnostic guest payload delivery

- **Status:** Proposed
- **Date:** 2026-06-06

## Context

The `export!` shims (ADR-0024) make the guest contract binding-agnostic *by string*: the host resolves a guest by probing export names (`receive_p32`, `init_p32`, `kind_scratch_reserve_p32`, …), so any language that compiles to wasm and exports those symbols can supply a guest. The host→guest *payload-delivery* path breaks that promise — it bakes the Rust/LLVM memory layout into the ABI in two places (issue #1395).

**The fast path writes to a fixed offset in the shadow stack.** Small mail, init config (ADR-0090), and rehydrate state (ADR-0016) are written straight to fixed low offsets in the guest's linear memory — `MAIL_OFFSET = 1024`, `STATE_OFFSET = 8192`, `CONFIG_OFFSET = 16384` — which sit inside the shadow-stack region of the standard stack-first wasm32 layout (a 1 MiB shadow stack at `[0, 1 MiB)`, then static data, then the heap). The size guard is a flat cap, `MAX_MAIL_PAYLOAD_BYTES = (1 << 20) − MAIL_SCRATCH_STACK_RESERVE − MAIL_OFFSET`, that assumes precisely that layout. #1337 proposed validating the layout against the guest's exported `__stack_pointer` / `__heap_base`; what shipped is the flat-cap fallback — "Rust guests don't export `__stack_pointer`," per the const's own doc — which bakes the layout in silently. A guest with the stack elsewhere, a non-default stack size, or `--stack-first` off would be clobbered with no diagnostic.

**The over-cap path is an aether-shaped scratch, not a written-down contract.** A payload above the flat cap rides `kind_scratch_reserve_p32` (`crates/aether-actor/src/ffi/mod.rs`) — one persistent guest-heap buffer, grown to a high-water mark and reused, never freed. That is correct: it exploits the run-token model, where exactly one inbound payload is live at a time and is consumed synchronously inside `receive` / `init`, so there is nothing to free. But it is an aether-specific narrowing that a portable binding author wouldn't expect, it isn't specified as a deliberate ABI primitive, and the small-payload path doesn't use it — so the two paths share no contract.

This is forward-looking ABI hardening. Today's Rust guest is correct and #1337's crash-safety work shipped; nothing here fixes a present break. The goal is to make the delivery primitive layout-independent and specify it as a contract a second binding can implement against, so a non-Rust guest — or the Component Model's `cabi_realloc` — slots in without footguns.

## Decision

Deliver every host→guest payload through a **single, generic guest allocator export** — `realloc`-shaped, aligned with the Component Model's `cabi_realloc` — and obtain the destination region from it. There is no host-chosen offset and no purpose-named scratch primitive at the boundary: the guest owns its memory, and the host writes into regions the allocator hands back. The SMALL/LARGE distinction is a host-side reuse pattern over that one allocator, and the scratch lives *underneath* the allocator as a guest-internal optimization.

1. **One allocator, the standard contract.** The guest exports a single allocator with the ordinary `cabi_realloc` shape — allocate, grow (possibly relocating), free — and the ordinary contract: two live allocations occupy non-overlapping regions, a grow returns the region's current pointer, and the caller owns what it gets back. This one function is the whole host↔guest delivery surface; there is no `…_small` / `…_large` family. Because the host writes only into regions the guest allocated, nothing lands at a host-chosen offset and nothing couples to the stack-first layout — assumption (1) is removed by construction.

2. **The host obtains two regions from that allocator and reuses them (run-token discipline).** At instantiate the host makes one small allocation, caches its pointer, and writes every payload that fits into it directly — the hot path (a tick, a key, most kinds — tens of bytes) is a plain `Memory::write` to the cached offset with no further allocator call. For a payload over that floor the host grows a second region via `realloc`, caching the returned pointer; it settles at the high-water mark of the largest payload that instance receives and is reused thereafter. "SMALL" and "LARGE" name these two host-cached regions — a reuse pattern, not two ABI functions.

3. **Independence is the load-bearing guarantee.** A grow of the large region may relocate it, but the allocator contract keeps it from overlapping the live small region, so the cached small pointer stays valid across any number of large grows. That is the whole reason the host can cache the hot-path pointer once and never re-fetch it; the large pointer, which may move, is taken from each `realloc`'s return value rather than cached.

4. **The scratch lives underneath the allocator, as a guest-side optimization.** A wasm/Rust guest may satisfy the first sub-threshold allocation from a fixed, reused scratch region instead of its general heap — the host's first small `alloc` returns the scratch, and because the run-token usage keeps only one such region live at a time, nothing else ever collides with it. This is invisible above the export boundary: the host calls the same allocator regardless. A guest that doesn't optimize — a plain allocator, or a Component-Model guest's stock `cabi_realloc` — is equally conforming. The scratch is a per-binding implementation choice, never part of the contract, and not required for the path to be fast (the host-side cache already makes the hot path call-free).

5. **One delivery path for every lifecycle payload.** Inbound mail, init config (ADR-0090), and rehydrate state (ADR-0016) are host→guest payloads delivered at different, mutually-disjoint lifecycle points. They all go through the same allocator and the same two cached regions — the small one if they fit, the grown one otherwise. The three fixed offsets (`MAIL_OFFSET` / `STATE_OFFSET` / `CONFIG_OFFSET`) and their three flat caps collapse into this single path.

6. **The host never frees; the guest never shrinks.** wasm linear memory only grows — there is no `memory.shrink`, so the free form (`realloc` to size 0) would return a region to the allocator's free list without handing pages back to the OS. The process footprint is the linear-memory high-water mark regardless, so freeing would pay allocation churn to reclaim nothing. The host therefore never calls free: the small region is allocated once and reused, the large region grown to high-water and reused. Per-instance delivery footprint is the small floor (kilobytes, always) plus the high-water of that instance's largest payload (zero if it never exceeds the floor). This is the wasm floor; no scheme returns those pages. Recorded here so the never-free discipline reads as deliberate rather than as an omission — `free` stays in the contract because a portable allocator carries it, but delivery never invokes it.

7. **The run-token usage invariant is part of the contract.** At most one small and one large payload are live at a time, each consumed synchronously inside the guest entry point before the host delivers the next. This is what licenses point 4's scratch optimization, and it is documented alongside the allocator contract so a second binding knows it may reuse delivery regions rather than treat each delivery as an independent, freeable allocation.

8. **Crash-safety is preserved.** A payload past the absolute ceiling (`MAX_DELIVERABLE_MAIL_BYTES`, 64 MiB, matching the default frame cap) is dropped with a loud log instead of asking the guest to allocate a region that could exhaust its memory and trap — the failure handling #1337 shipped. Because the guest sizes and owns every region, there is no host write past the end of a guest buffer.

The change from today's surface: the fixed `MAIL_OFFSET` / `STATE_OFFSET` / `CONFIG_OFFSET` writes and their flat caps retire, and the delivery primitive becomes a generic guest allocator — a `cabi_realloc`-shaped allocate / grow / free export. Today's `kind_scratch_reserve_p32` (a one-argument grow-a-single-buffer call) is the narrowed form this replaces, not a primitive carried forward; the host gains an instantiate-time small allocation it caches. Exact export spelling and signature are pinned in the implementing PR — it is named as the generic allocator it is (`alloc` / `realloc`, `cabi_realloc`-aligned), with no `kind` or `scratch` in the name.

## Consequences

### Positive

- **The layout footgun is gone.** No host write lands in shadow-stack space, and no cap encodes the stack-first layout. A guest reports where its buffers are; any wasm-targeting language can supply one without reproducing Rust's `--stack-first` layout.
- **The delivery primitive is the canonical allocator, not a bespoke surface.** A single `cabi_realloc`-shaped export plus a documented usage invariant (small region cached and reused, large region grown, never freed, one payload of each class live at a time) is a contract a second binding implements against — and a stock `cabi_realloc` satisfies it directly, so the Component Model path is open rather than foreclosed.
- **One delivery path, three fewer offsets and caps.** Mail, config, and state stop being three special cases with three fixed offsets and three flat caps; they are one path through the allocator and two host-cached regions.
- **The hot path keeps its cost.** Small-payload delivery stays a single `Memory::write` to a cached offset — no per-payload guest call, no per-payload allocation — so the common case is no slower than today's fixed-offset write while no longer assuming the layout.
- **Footprint tracks usage.** An instance that only ever sees small mail pays a few kilobytes; LARGE is paid only by instances that actually receive large payloads, and only up to their own high-water.

### Negative

- **Guest SDK and host both change.** The Rust guest's `kind_scratch_reserve_p32` is replaced by a generic `cabi_realloc`-shaped allocator export — it may keep a reused scratch region underneath for the first small allocation, but that is an internal detail, not the export; the host gains an instantiate-time small allocation it caches and routes config/state/mail through the unified path. `Component::{instantiate, deliver}` and `call_on_rehydrate` all move onto it.
- **Prebuilt component wasm must be rebuilt.** The export surface changes, so components built against the old `kind_scratch` surface need a rebuild before they load — the same rebuild discipline a chassis-mailbox rename already requires.
- **The small-region floor is a tuning choice.** Too small and common-but-not-tiny mail spills to the grown region (a guest call where a cached write would do); too large and every instance over-pays the always-allocated floor. The threshold is a sizing decision the PR must justify, not an obvious constant.

### Neutral / forward

- **Supersedes the mechanism of #1337's flat cap while keeping its guarantee.** #1337 made oversize delivery crash-safe via the flat cap plus the reusable reserve buffer; this ADR removes the flat cap's layout assumption by removing the fixed-offset write entirely, and preserves crash-safety through the LARGE cap + guest-owned sizing. #1337 is the safety layer; this is the binding-agnosticism layer it left behind.
- **Extends the guest ABI set** (ADR-0024 pointer ABI, ADR-0028 kind manifest, ADR-0033 dispatch) with the delivery contract those ADRs assumed but never specified.
- **A future non-Rust or Component-Model guest is the motivating consumer**, not a committed one — this is roadmap-paving so the FFI is intuitive and portable when a second binding becomes real.

## Alternatives considered

- **Document + validate the stack-first layout, keep both paths (issue direction A).** Read the guest's `__stack_pointer` / `__heap_base` at instantiate, validate the layout, and keep the fixed-offset fast path as a documented floor. Rejected: Rust guests don't export `__stack_pointer`, so the validation degrades to the flat-cap fallback for the only guest that exists — it documents the assumption without removing it, and keeps two delivery paths instead of one.
- **Allocator-only delivery with a free per payload (issue direction B, naive).** Always deliver through `cabi_realloc` / `malloc`, allocating and freeing a fresh region per payload. Rejected as stated: the per-payload allocation is a cost of *calling the allocator per payload*, not of allocator-based delivery — and the free reclaims nothing under wasm's no-shrink memory. Allocating SMALL once and reusing it (this ADR) keeps the general-allocator contract while paying neither cost.
- **Single shared buffer for both small and large (issue direction C, one region).** Keep the no-free reused buffer but make it the only region, including for small payloads. Rejected: one shared region means a large-payload grow moves the region a small payload would reuse, forcing the host to re-fetch the pointer (a guest call) on the hot path. Holding the small allocation separately keeps the cached small pointer stable across large grows; the host keeps two regions while the boundary stays one allocator.
- **Keep `kind_scratch` as the MVP and do nothing.** Rejected for the roadmap, accepted for the present: the current scheme is correct for the Rust guest and is fine until a second binding or the Component Model becomes real. This ADR is the plan for that point, not a fix for today.
