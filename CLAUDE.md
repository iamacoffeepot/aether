# CLAUDE.md

This file provides guidance to Claude Code (claude.ai/code) when working with code in this repository.

## Status

Pre-1.0 Rust project (edition 2024). Vision: a game engine where Claude sits in a harness as assistant/engineer/designer. Architectural direction (see `docs/adr/`): a thin native **substrate** owns I/O, GPU, audio, and hosts a WASM runtime; engine **components** run as WASM modules and communicate via a **mail** system. (The whole system — substrate + components + tooling — is "Aether" or "the engine"; the substrate is just the native base layer.)

## Infrastructure crates (ADR-0069)

The non-component infrastructure cluster has three crates with role-distinct names:

- **`aether-data`** — universal data layer. Typed-id newtypes (`MailboxId`, `KindId`, `HandleId`), wire identity types (`EngineId`, `SessionToken`, `Uuid`), schema vocabulary (`SchemaType`, `KindShape`, `KindLabels`, `InputsRecord`, canonical bytes), `Kind` / `Schema` / `CastEligible` traits, `Ref<K>`, encode/decode helpers, native descriptor inventory. `no_std` + `alloc`. Every consumer that describes typed bytes depends on this.
- **`aether-codec`** — byte-encoding toolkit (formerly `aether-params-codec`). Pure functions: schema-driven JSON ↔ wire bytes over `aether_data::SchemaType` (`encode_schema`, `decode_schema`) plus generic length-prefix postcard stream framing (`frame::encode_frame` / `read_frame` / `write_frame`, ADR-0072). Future home for non-JSON formats (msgpack, save-format adapters) and non-postcard framing variants.
- **`aether-kinds`** — substrate vocabulary: concrete kind types (`Tick`, `Key`, `DrawTriangle`, `aether.audio.*`, `aether.fs.*`, `aether.component.*`, `aether.input.*`, `aether.actor.monitor_notice`, `aether.camera`). (Pre-#613 there was a typed-`MailboxId` registry under `mailboxes::*`; that retired in favour of inline `mailbox_id_from_name(<Cap>::NAMESPACE)` at use sites. Issue #775 retired `HUB_BROADCAST_MAILBOX_NAME` and the rest of the `aether.observation.*` family — `frame_stats`, `component_died`, `substrate_dying` — alongside the `BroadcastCapability` MCP fan-out; `MonitorNotice` (the framework close-notification kind, ADR-0079) survived and was rehomed to `aether.actor.*` by issue #779 since it was never observation, just a v1 squatter in that namespace.)

The hub channel wire vocabulary (`EngineToHub`, `HubToEngine`, `Hello`, `MailFrame`, etc.) lives in `aether-substrate-bundle::hub::wire` next to the runtime that speaks it (ADR-0072). Future sibling stream protocols define their own frame-enum modules in their own crates and reuse `aether-codec::frame` for the generic helpers.

The chassis layer consolidated in May 2026 (ADR-0073). All four chassis live in a single multi-bin `aether-substrate-bundle` crate: `src/{desktop,headless,hub,test_bench}/` submodules with matching `src/bin/{desktop,headless,hub,test-bench}.rs` entry points (and the hub library at `src/hub/` housing the substrate-side client, wire types, and MCP coordinator). The shared substrate runtime is `aether-substrate` (renamed from `aether-substrate-core` to match the workspace `X` / `X-{suffix}` convention). The in-process `TestBench` library API surfaces at `aether_substrate_bundle::test_bench::TestBench` for component, scenario, and demo consumers.

`aether-id`, `aether-mail`, `aether-params-codec`, and `aether-hub-protocol` are retired. `Mail<'_>` and `Sink<K>` live in `aether-component` (guest SDK); the host's dispatcher-side `Mail` lives in `aether-substrate`. The "transport envelope" abstraction is split between SDK and dispatcher rather than shared.

## Workflow

- **Exploration and design discussion** happens in chat with the user. No artifact required.
- **Planned work** (spikes, features, open investigations) lives in GitHub Issues. Referenced by the PR that closes them.
- **Load-bearing architectural decisions** are recorded as ADRs in `docs/adr/NNNN-title.md`. Use `docs/adr/TEMPLATE.md` when starting a new one. Number sequentially. An ADR is reviewed via a PR like any other change.
- **Branches**: `type/short-slug` (e.g. `chore/ci-bootstrap`, `feat/mail-runtime`, `docs/adr-workflow`).
- **Commits and PR titles** follow Conventional Commits (`type(scope): subject`). Enforced in CI against PR titles. Main uses squash-merge with PR title as the commit subject, so PR title quality matters.
- **Merging**: `main` is protected (PR required, all CI checks required, linear history, no force-push). Claude does not push to `main`, does not force-push reviewed branches, does not self-merge, and asks before destructive operations.
- **PRs** should be small and focused — one concept per PR.
- **Recursion in load-bearing code**: prefer iterative implementations (explicit work-stack/queue, arena-with-indices for tree data) over recursive ones in any algorithm whose depth could exceed a few hundred frames in practice. Recursion is OK for parse/AST walks where depth is structurally bounded by a small input file. Either way, recursive code on user-controlled or geometrically-derived data must enforce a depth/budget cap that returns an error rather than overflowing the stack.

## MCP harness

Claude drives a running engine through MCP — the concrete form of the "Claude-in-harness" vision.

**The harness is the out-of-process `aether-mcp` crate (issue 763 P5).** It's an RPC client — it dials the hub's `RpcServerCapability` and relays each tool call as a wire `Call`; the hub binary itself holds no MCP code. The dev workflow is two processes:

```
cargo run -p aether-substrate-bundle --bin aether-substrate-hub   # RPC server on 127.0.0.1:8901
cargo run -p aether-mcp                                          # serves MCP HTTP on 127.0.0.1:8890
```

`.mcp.json` points Claude Code at `:8890`. The hub boots its RPC server unconditionally (`AETHER_RPC_PORT` overrides `8901`); `aether-mcp` dials it (`AETHER_HUB_RPC_ADDR` overrides) and serves MCP on `AETHER_MCP_PORT` (default `8890`). Get a substrate with `spawn_substrate` — the hub forks it, assigns its RPC port, and connects a per-engine proxy; `aether-mcp` tracks the fleet through the hub's `aether.engine` cap. (Issue 763 P5f collapsed the hub to a thin chassis hosting `RpcServerCapability` + `EngineServer` + `TraceObserverCapability`; the legacy `EngineToHub` engine listener, hub-side sessions, `ProcessCapability`, loopback drainers, and embedded MCP server are gone.)

Tools (`mcp__aether-hub__*`):

- `list_engines` — every engine the hub supervises: `{engine_id, rpc_port}`.
- `spawn_substrate(binary_path, args?)` — fork+exec a substrate; the hub assigns its RPC port (injected as `AETHER_RPC_PORT`) and connects a proxy. Returns `{engine_id, rpc_port}`.
- `terminate_substrate(engine_id)` — the hub forwards to the engine's proxy, which SIGKILLs the child and self-shuts-down.
- `send_mail(mails)` — best-effort batch. Each item — `{engine_id, recipient_name, kind_name, params}` — is schema-encoded against the substrate kind vocabulary and routed to the engine as a wire `Call`. Per-item status; one failure doesn't abort siblings.
- `describe_kinds` — the substrate kind vocabulary with full schemas (the static inventory `aether-mcp` ships with — enough to build `send_mail` params; component-defined kinds aren't included).
- `describe_component(engine_id, mailbox_id)` — a loaded component's receive-side capabilities (ADR-0033): handler kinds + per-handler docs, `#[fallback]` presence, component doc. Reads `aether-mcp`'s component cache, populated by `load_component` / `replace_component`.
- `load_component(engine_id, binary_path, name?)` — `aether-mcp` reads the wasm and forwards `aether.component.load` to the engine's `aether.component` mailbox, awaiting `LoadResult`. The component's kind vocabulary rides in the wasm's `aether.kinds` custom section (ADR-0028).
- `replace_component(engine_id, mailbox_id, binary_path, drain_timeout_ms?)` — ADR-0022 freeze-drain-swap; forwards `aether.component.replace`, awaits `ReplaceResult`. `drain_timeout_ms` is accepted for wire compatibility but ignored (post-ADR-0038 the splice is structural).
- `capture_frame(engine_id, mails?, after_mails?)` — synchronous frame readback as a PNG, returned inline as image content. `mails` dispatch before readback (state changes that should appear in the frame), `after_mails` after (cleanup); a bad bundle entry aborts the whole capture.
- `engine_logs(engine_id, max?, level?, since?)` — pull recent log entries from a substrate's ring (issue 776, restoring ADR-0023 §4 under the forward model). `max` defaults to 100 and clamps to 1000; `level` filters server-side (`"trace"`|`"debug"`|`"info"`|`"warn"`|`"error"`); `since` is the prior call's `next_since` for cursor-paginated polling. Response: `{engine_id, entries: [{timestamp_unix_ms, level, target, message, sequence, origin}], next_since, truncated_before}`. `truncated_before` is `Some(seq)` when the ring evicted entries the caller hadn't seen yet — a signal to poll more often or accept the gap. Only in-actor `tracing::*` events flow into the ring; host-emitted events (substrate boot, scheduler, panic hook) go to stderr only per issue #601. Filter via `AETHER_LOG_FILTER` (`tracing_subscriber::EnvFilter` syntax; default `info`).

`receive_mail` retired with the broadcast cap (issue 775); replacement is parked under "future user-space TCP / websocket observer."

When verifying substrate behavior end-to-end, reach for the MCP harness before running a new test binary.

## Scenario runner (ADR-0067)

For Rust integration tests and CI gating that don't need a live MCP session, the **scenario runner** drives the same in-process substrate from a Rust thread. The surface lives in three places:

- **`aether_substrate_bundle::test_bench`** (lib module of `aether-substrate-bundle`, ADR-0073) — `TestBench::start()` boots a full chassis (scheduler, mail queue, wgpu offscreen render target) on the test thread, with a loopback channel attached to outbound so substrate replies route back without going through a hub. `advance(ticks)`, `capture()`, `send_mail<K>()`, `send_bytes(name, kind_id, bytes)`.
- **`aether-scenario`** — declarative `Script` of `Step`s (`Advance`, `Capture`, `Assert`, `LoadComponent`, `SendMail`) parsed from YAML. `Runner::run(&mut bench, &script)` walks the steps, encoding `SendMail` params via `aether-codec::encode_schema` against `aether_kinds::descriptors::all()` — same path the hub uses for `mcp__aether-hub__send_mail`, so any substrate kind is sendable from a script for free. Returns a `RunReport` with per-step pass/fail. Ships with an `aether-scenario` bin (`cargo run -p aether-scenario --bin aether-scenario <path.yml>`) — boots a TestBench, runs the script, prints a per-step report, exits 0/1. The agent-facing entry point.
- **`aether-scenario-macros`** — `aether_scenario::scenario_dir!("scenarios")` proc-macro emits one `#[test]` per `.yml` file under the named directory. Component crates author scenarios as plain YAML and get IDE-runnable tests for free. Stays separate per Rust's `proc-macro = true` requirement.

Scenarios need a wgpu adapter; CI installs `mesa-vulkan-drivers` on Linux runners and pre-builds component wasm before `cargo test`. Driverless dev boxes skip cleanly with an `eprintln!`. Reach for the scenario runner over a hand-rolled test binary when you want repeatable substrate verification in `cargo test`; reach for the MCP harness when you want exploratory or live observation.

Wasm components are discovered in CI as `cargo metadata` packages with `crate-type = cdylib` AND a dep on `aether-component` (issue 439). Both signals are structural — no filename convention to remember. Forgetting either signal means the crate isn't a component anyway.

The observation path (ADR-0008) retired in issue #775 alongside the embedded MCP server: with no fan-out target left, `BroadcastCapability`, the `hub.claude.broadcast` mailbox, and the `aether.observation.frame_stats` / `component_died` / `substrate_dying` kinds were removed. Components still reply to a specific sender (Claude session or other component) via the `reply_mail` host fn (ADR-0013, ADR-0017), wrapped by `Ctx::reply` in the SDK; a future user-space observer (TCP, websocket, or session-targeted mail) is the path forward for engine-out fan-out.

Input streams (tick, key, mouse_move, mouse_button) are publish/subscribe (ADR-0021): the substrate boots with empty subscriber sets and drops every input event until something subscribes. To wire a freshly-loaded component into the platform's event stream, mail `aether.input.subscribe` with `{ "stream": "Tick" | "Key" | "MouseMove" | "MouseButton", "mailbox": <id from load_result> }` to the `aether.input` mailbox. Multiple components may subscribe to the same stream; the substrate fans out to every subscriber. `aether.input.unsubscribe` removes one. Both reply via `aether.input.subscribe_result`. Subscriptions are auto-cleared when a component is dropped and preserved across `replace_component` (the mailbox id is stable). (Pre-issue-638 Phase 2 these kinds rode `aether.control.*` on the component-host cap; Phase 2 rehomed them to a dedicated `InputCapability`.)

`aether.component.replace` is a structural splice (ADR-0022 + ADR-0038): the substrate splices the mailbox's inbox over to a new dispatcher, drops the old sender, and joins the old dispatcher once it drains naturally. Mail that arrives during the splice lands in the new dispatcher's inbox. The `drain_timeout_ms` field is accepted on the wire but currently ignored — post-ADR-0038 there is no freeze-drain timeout to enforce. A future ADR may repurpose it as a join-timeout if stuck guests (e.g. a wasm `deliver` in a CPU-only loop) become common.

Window + platform state is mailed, not MCP-toolled. `aether.control.platform_info` returns a `PlatformInfoResult` snapshot (OS + arch, engine version + kind count, GPU adapter/limits, full monitor list with available video modes, current window mode + size). (The wire also carries an `EngineInfo.workers` field; post-ADR-0038 there is no shared worker pool — one OS thread per component — so the value is retained for wire stability but no longer reflects scheduler parallelism.) `aether.control.set_window_mode` switches the window at runtime between `Windowed { width?, height? }`, `FullscreenBorderless`, and `FullscreenExclusive { width, height, refresh_mhz }`; the substrate resolves exclusive modes against the window's current monitor and replies with the actually-applied size. `aether.control.set_window_title` updates the window title; the reply echoes the applied title back. Boot-time overrides: set `AETHER_WINDOW_MODE` to `windowed`, `windowed:WxH`, `fullscreen-borderless`, or `exclusive:WxH@HZ` (integer Hz) to pick the initial mode before any component loads, and `AETHER_WINDOW_TITLE` to set the initial title (default `aether`). `platform_info`, `set_window_mode`, `set_window_title`, and `capture_frame` are desktop-chassis-only; the headless chassis replies `Err { error: "unsupported on headless chassis…" }` (ADR-0035) so callers fail fast instead of hanging on a reply that never comes.

The headless chassis (`aether-substrate-bundle::headless`, ADR-0035 Phase 3) drives ticks from a std timer instead of winit vsync; cadence defaults to 60 Hz and is overridable via `AETHER_TICK_HZ`. It inherits the same hub client, mail scheduler, and `aether.component.{load,drop,replace}` surface as desktop, so components (and the dev workflow of loading-via-MCP) port across chassis unchanged. A nop `aether.render` mailbox is pre-registered so desktop-designed components that emit `DrawTriangle` and `aether.camera` every tick don't warn-storm `engine_logs` (post-ADR-0074 §Decision 7 the camera mailbox folded into render, so one nop mailbox covers both kinds).

Vertex geometry is world-space. The substrate holds a 4×4 `view_proj` uniform (column-major, 64 bytes) that the vertex shader applies to every vertex: `clip_pos = camera.view_proj * vec4(position, 1.0)`. The uniform defaults to identity, so components that still emit clip-space-ish world coords render unchanged until a camera component starts publishing. Camera components send `aether.camera { view_proj: [f32; 16] }` to the desktop chassis's `aether.render` mailbox (the camera mailbox folded into render per ADR-0074 §Decision 7; the kind name `aether.camera` is unchanged) — latest value wins, uploaded to the GPU uniform before each draw. Fire-and-forget: no reply. `Vertex` grew a `z: f32` field (24 bytes on the wire; `DrawTriangle` is 72 bytes per triangle). The reference camera implementation is `aether-camera`, a multi-camera/multi-mode component: it hosts N named cameras (each in `Orbit` or `Topdown` mode) and publishes only the active one's `view_proj` per tick. Boots with one default camera (`name: "main"`, orbit) marked active, so loading the component immediately yields a visible camera (~12 s/revolution auto-orbit, matching the prior single-mode behaviour). Driver surface: `aether.camera.create { name, mode: ModeInit }`, `aether.camera.destroy { name }`, `aether.camera.set_active { name }`, `aether.camera.set_mode { name, mode }`, and per-mode delta kinds `aether.camera.orbit.set { name, params: OrbitParams }` and `aether.camera.topdown.set { name, params: TopdownParams }`. `OrbitParams` / `TopdownParams` use `Option<...>` per field, so the same struct seeds creation (compiled defaults fill unspecified fields) and streams partial deltas — `OrbitParams { distance: Some(5.0), ..None }` zooms without disturbing yaw / pitch / target. Inactive cameras still tick (orbit yaw keeps accumulating) so re-activating doesn't snap. Mode mismatch on `*.set` warns and no-ops; switch with `set_mode` first. Per ADR-0066 (amended by issue 552 stage 1.5) the kind types, parameter structs, and runtime live together in the dual-output `aether-camera` crate; other components depend on `aether-camera` for the wire shape, and the same crate emits the cdylib the substrate loads.

Vector and matrix math lives in the `aether-math` crate: `Vec2`, `Vec3`, `Vec4`, `Mat4`, `Quat`, `Aabb` (column-major, YXZ Euler, right-handed Y-up, wgpu `[0, 1]` clip-space depth, scalar `f32`, `no_std`). Reach for it before writing inline `cross` / `dot` / `normalize` / `rotate` / aabb checks. When something's missing, add it to `aether-math` rather than reinventing locally — domain-agnostic spatial/numeric primitives only (mesh, render, CSG, audio types stay in their owning crate). `[f32; 3]` survives at wire boundaries (`Vertex`, AST nodes, the OBJ exporter) and bridges via `Vec3::from_array` / `to_array`; everything in between should use the typed forms.

Depth testing is on in the desktop chassis: a `Depth32Float` texture paired with the offscreen color target, cleared to 1.0 each frame, with `LessEqual` comparison. Combined with the topdown camera (which looks down `-Z`) this means larger world-z draws on top of smaller world-z — the intuitive "push things up to bring them forward" convention. Floors / backdrop geometry sit at `z = 0`; movers / foreground should sit at `z = 0.1` or higher. The reference `aether-player-component` draws its body at `z = 0.1`.

Mesh authoring goes through the mesh viewer component (the dual-output `aether-mesh-viewer` crate, with `aether.mesh.load` and any future driver kinds at the crate root per ADR-0066 amended by issue 552 stage 1.5), a thin file-loader for both the s-expression mesh DSL (ADR-0026 + ADR-0051) and Wavefront OBJ. Load the wasm with `load_component` and pick whatever name fits the role — the bare names `render` / `camera` / `audio` / `fs` / `net` / `handle` / `log` are non-reserved (chassis-owned mailboxes use the `aether.<name>` namespace per ADR-0074 Phase 5, so a user-space component loaded as `"render"` doesn't collide). Drive the loaded viewer with one mail kind: `aether.mesh.load { namespace, path }` asks the viewer to fetch bytes through the `aether.fs` mailbox (ADR-0041 namespaces) and parse them on reply, dispatching on extension — `.dsl` runs through `aether-mesh`'s parser+mesher and emits polygon-edge wireframes alongside filled triangles, `.obj` runs through a built-in fan-triangulation parser with a default soft-blue color and no wireframe. The DSL agent loop is now write-file-then-load: emit the DSL via `aether.fs.write { namespace, path, bytes }`, then `aether.mesh.load` against the same path. Fire-and-forget. Each successful load parses, meshes, and atomically replaces the cached triangle list; partial parse / mesh failure keeps the previous mesh visible and surfaces the error via `engine_logs`. Every tick the viewer replays the cached triangles to the `aether.render` mailbox, so once a mesh is loaded it stays on-screen with no further mail. Colors are DSL palette indices (`:color N`, `u32`) mapped via an 8-entry built-in palette wrapped modulo length. Vocabulary covers cube / cylinder / cone / sphere / wedge primitives, `extrude` for face extrusion, `mirror` / `array` for repetition, and `sweep` for path-driven prism generation (parallel-transport framing is normative — ADR-0051). Examples live in `crates/aether-mesh/examples/` (box, lamp_post, teapot). The crate `aether-mesh` is library-only (parser + mesher); the viewer component is the substrate-side wrapper. Boolean composition (`union` / `intersection` / `difference`) was retired by ADR-0062 and lives on the `archive/csg-bsp` branch.

File I/O is a mail-based mailbox (ADR-0041). The substrate owns an `aether.fs` mailbox and an `AdapterRegistry` mapping logical namespaces to storage backends. v1 ships three namespaces — `save://` (writable, per-user persistent — default `dirs::data_dir()/aether/save`), `assets://` (read-only — default `{current_exe}/../assets`), `config://` (writable — default `dirs::config_dir()/aether`) — each with a `LocalFileAdapter` that normalizes paths (`..` / absolute prefixes → `Forbidden`) and writes atomically via tmp+rename. Components mail `aether.fs.read { namespace, path }` / `write { namespace, path, bytes }` / `delete { namespace, path }` / `list { namespace, prefix }` to the `aether.fs` mailbox (the kind namespace `aether.fs.*` is the kind id namespace, distinct from the mailbox name even though both share the `aether.fs` prefix). The mailbox was renamed from `aether.io` (issue #633) — `fs` matches the `std::fs` muscle memory and disambiguates from the network IO sinks (`aether.tcp`, `aether.http`); the ADR-0041 design (namespaced byte storage with pluggable adapters) is unchanged. Each reply kind echoes the originating `namespace` + `path` (or `prefix` for `List`) on both `Ok` and `Err` variants so components correlate reply-to-request by the echoed fields + the reply kind itself — no pending-op queue, no FIFO dependency. `WriteResult` omits `bytes` from the echo so a megabyte write still produces a small reply. On malformed-request decode failure the echo collapses to empty strings and `FsError::AdapterError` carries the diagnostic. `FsError` is one of `NotFound` / `Forbidden` / `UnknownNamespace` / `AdapterError(String)`. Namespace in the mail is the short prefix (`"save"`), not `"save://"` — the double-colon form is a UX convention in docs and logs. Desktop and headless chassis both wire the mailbox; hub chassis doesn't (mail to `aether.fs` warn-drops as unknown mailbox). Env-var config: `AETHER_SAVE_DIR`, `AETHER_ASSETS_DIR`, `AETHER_CONFIG_DIR` override the defaults. Adapter calls run synchronously on the dispatcher thread — fine for KB-MB save/config files; asset streaming would want a host-fn fast path, deferred. TOML + CLI layers in the precedence stack (CLI > env > TOML > defaults) are committed but not shipped in v1.

Audio is a desktop-chassis peripheral (ADR-0039). The substrate owns a `cpal` output stream fed by a hand-rolled synth: one voice per `(sender_mailbox, instrument_id, pitch)` triple, ADSR envelopes, flat mix scaled by master gain, soft-clipped via `tanh`. v1 ships five built-in instruments (`sine_lead`, `square_bass`, `triangle`, `saw_lead`, `pluck`) indexed by `NoteOn.instrument_id`; runtime-defined patches are parked. Components emit `aether.audio.note_on { pitch, velocity, instrument_id }` and `aether.audio.note_off { pitch, instrument_id }` to the `aether.audio` mailbox — fire-and-forget cast kinds; the voice-key keeps notes from different senders independent. `aether.audio.set_master_gain { gain }` also flows through the `aether.audio` mailbox and replies with `SetMasterGainResult::Ok { applied_gain }` on desktop or `Err { error }` on headless / hub / audio-disabled substrates. Mixing and effects are user-space composition — the substrate sums voices flat, so a music component that wants ducking or buses forwards events through its own mail surface. Env vars: `AETHER_AUDIO_DISABLE=1` skips cpal init (nop sink, `SetMasterGain` replies Err); `AETHER_AUDIO_SAMPLE_RATE=<hz>` requests a specific rate (default: device preference, typically 44100 or 48000).

**Recipient-name convention.** Chassis-owned mailboxes live under the `aether.<name>` namespace (ADR-0074 Phase 5 retired the `aether.sink.*` infix that ADR-0058 introduced) — `"aether.render"`, `"aether.audio"`, `"aether.fs"`, `"aether.http"`, `"aether.handle"`, `"aether.log"`, `"aether.input"` for input subscriptions, plus `"aether.component"` for the component load surface (renamed from `"aether.control"` in issue 638 phase 3; drop and replace are forwarded from the cap to each component's trampoline). Pre-Phase-3 there was a separate `"aether.sink.camera"`; ADR-0074 §Decision 7 folded the camera mailbox into render, so `aether.camera` mail now addresses `"aether.render"` (the kind name `aether.camera` is unchanged). Loaded wasm components live at `"aether.component.trampoline:NAME"` (issue 634 Phase 4): each `WasmTrampoline` is an instanced `NativeActor` under that namespace prefix, with `NAME` the agent-supplied name (or the wasm-declared default). `LoadResult.name` returns the full address; agents send subsequent peer/runtime mail to that string. Bare component names (`"camera"`, `"player"`, etc.) are not registered in the registry — addressing a loaded component by its bare name warn-drops as unknown. Inside the hub's MCP `send_mail`, `recipient_name` names the mailbox; `kind_name` names the payload shape. Mailbox name and kind namespace happen to share a leading prefix (e.g. `aether.audio` vs `aether.audio.note_on`) but are routed independently — `recipient_name` is the mailbox lookup, `kind_name` is the payload schema. Send `aether.audio.note_on` (kind) to `aether.audio` (mailbox), `aether.draw_triangle` (kind) to `aether.render` (mailbox), or `aether.camera.destroy` (kind) to `aether.component.trampoline:cam` (the loaded camera component's trampoline).

Design detail lives in ADR-0006 (wire + topology), ADR-0007 (schema-driven encoding), ADR-0008 (observation path), ADR-0009 (hub-supervised substrate spawn), ADR-0020 (symmetric receive_mail decode), ADR-0021 (input stream subscriptions), ADR-0022 (drain-on-swap for replace_component), ADR-0023 (substrate log capture + engine_logs), ADR-0024 (`_p32`-suffixed FFI in anticipation of wasm64), ADR-0029 (name-derived 64-bit MailboxIds), ADR-0030 (schema-hashed 64-bit kind ids — `Kind::ID = fnv1a_64(KIND_DOMAIN ++ canonical(name, schema))`, `resolve_kind_p32` retired; `MailboxId` uses the same FNV-1a 64 with a disjoint `MAILBOX_DOMAIN` prefix per issue #186; the SDK auto-subscribe walker described in ADR-0030 Phase 2 retired in issue #403, and the cap-side manifest auto-subscribe retired in issue 640 — components subscribe explicitly from `wire`), ADR-0031 (const-constructible schema representation), ADR-0032 (canonical schema bytes + labels sidecar — `aether.kinds` v0x04 + `aether.kinds.labels` v0x03 dual-section manifest; v0x04 retired the v0x03 trailing `is_stream` byte alongside the auto-subscribe path per issue 640), ADR-0033 (handler-driven inputs manifest — `#[handlers]` attribute macro, `aether.kinds.inputs` wasm custom section, `describe_component` MCP tool; The component FFI surface uses a `_p32` suffix on every pointer-typed import (`aether::send_mail_p32`, `reply_mail_p32`, `save_state_p32`) and on the `receive_p32` / `on_rehydrate_p32` exports. Non-pointer exports (`init`, `on_replace`, `on_drop`) are unsuffixed. The suffix locks the wasm32/wasm64 naming convention without committing to dual registration today — see ADR-0024 for the deferred Phase 2. `send_mail_p32`'s `recipient` and `kind` are scalar `i64` / `u64` — ADR-0029 made mailbox ids 64-bit name hashes and ADR-0030 made kind ids 64-bit schema hashes; neither falls under the `_p32` (pointer-width) convention. `resolve_mailbox_p32` was retired in ADR-0029; `resolve_kind_p32` was retired in ADR-0030 Phase 2. Guests get `Kind::ID` and `mailbox_id_from_name` as compile-time consts — no host-fn round trips for address resolution. `init` and `wire` both receive a `mailbox_id: u64` parameter so the FFI ctx can self-address — the SDK's init walker (ADR-0030 Phase 2) retired in issue #403, but the parameter stays useful: components subscribe explicitly from `wire` by mailing `aether.input.subscribe { kind, mailbox: ctx.mailbox_id() }` to `InputCapability` (issue 640 Phase 2).

Components declare their receive-side kind vocabulary with `#[handlers]` on a single `impl Component for C` block (ADR-0033). Each handled kind is a `#[handler] fn on_whatever(&mut self, ctx: &mut Ctx<'_>, mail: K)` method — the decoded payload arrives as the third parameter, so the kind is inferred from the signature and the author never writes a typelist or an `is::<K>()` branch. An optional `#[fallback] fn on_anything(&mut self, ctx, mail: Mail<'_>)` catches everything else; omit it for a strict receiver. The macro codegens a dispatch table and emits an `aether.kinds.inputs` wasm custom section enumerating every handler (id + name + per-handler rustdoc extracted from `# Agent` sections) — that's what `describe_component` surfaces over MCP, so Claude can ask "what does this component listen for?" without running it. `Sink<K>` keeps its explicit field for sends (mailbox names are data, not types). ADR-0027's `type Kinds` typelist and the generic `Component::receive` dispatch method are retired — handlers are the sole source of truth.

## Commands

- Build: `cargo build` (release: `cargo build --release`)
- Run: `cargo run -p <crate>` — workspace root has no default binary. The chassis binaries all live in `aether-substrate-bundle`; pick one with `--bin`: `cargo run -p aether-substrate-bundle --bin aether-substrate-hub`, `--bin aether-substrate` (desktop), or `--bin aether-substrate-headless`.
- Test: `cargo test` (single test: `cargo test <name>`; single-threaded with output: `cargo test -- --nocapture --test-threads=1`)
- Lint: `cargo clippy --all-targets -- -D warnings`
- Format: `cargo fmt` (check-only: `cargo fmt -- --check`)
- Type/borrow check only: `cargo check`

## Pre-push pre-flight

`scripts/preflight.sh` runs the CI-equivalent local checks (fmt + clippy + doc + nextest + wasm32 component cross-build) over the workspace. The pre-push git hook at `.githooks/pre-push` invokes it automatically against the changed-file set for each push, and stamps `.git/aether-preflight-passed` with the HEAD sha on success so a re-push of the same commit short-circuits.

Enable once per clone: `scripts/setup-githooks.sh` (sets `core.hooksPath -> .githooks`).

Exception classes — pushes that skip the Rust pre-flight:

- **Docs-only**: every changed path matches `docs/**` or `*.md` at the root.
- **CI / repo-config-only**: every changed path is `.github/**`, `.claude/**`, `.githooks/**`, `scripts/**`, `qodana.{yaml,sarif.json}`, `.mcp.json`, `.gitignore`, `.gitattributes`, or `{rust-toolchain,rustfmt,clippy}.toml`. CI itself runs on workflow changes and validates the workflow shape.
- A mixed push (e.g. one `.rs` + one `.md`) runs the full pre-flight; the exception only applies when every file matches the class.

A Claude-side hook (`.claude/hooks/check-pre-push.sh`) checks the stamp ahead of `git push` / `gh pr create`. If HEAD doesn't have a matching stamp, the hook blocks the push and prompts Claude to run `scripts/preflight.sh` plus `mcp__rustrover__get_file_problems` over each changed `.rs` file — the qodana-equivalent surface the shell-only git hook can't reach.

Bypass either layer with `git push --no-verify` (or omit the matching stamp deliberately and skip the Claude hook by tagging the override per the existing hook conventions).

## Qodana pre-flight (local)

Qodana gates merges via the `ci-pass` aggregator (iamacoffeepot/aether#941), but running `qodana scan` locally is blocked: the container's cargo-metadata pass times out on the colima virtiofs bind mount. The local equivalent is RustRover's inspection set — Qodana's `qodana.recommended` profile rehosts most of its checks from the same IntelliJ inspector RustRover runs in-IDE.

- **Per-file**: open the file; the gutter + Problems tool window surface the same inspection ids Qodana reports (`RsUnnecessaryParentheses`, `RsApproxConstant`, `DuplicatedCode`, etc.).
- **Whole-project**: `Code → Inspect Code…` → scope: `Whole project` (or `Uncommitted files` for a fast pre-push check) → profile: `Project Default`. Export the results via the inspection result toolbar if a diff against CI is needed.
- **Agent-side** (Claude): `mcp__rustrover__get_file_problems` returns the same problem set for one file (`errorsOnly: false` for the full qodana-equivalent surface). Iterate it over `git diff --name-only` before commits on Qodana-touched files (path-prefix sweeps, duplicated-code refactors, etc.) instead of waiting for the CI gate. RustRover MCP has no project-wide inspect runner — for whole-project pre-flight fall back to the IDE menu above.

Caveats: IDE inspections don't run Qodana's Cargo.toml-level checks (`NewCrateVersionAvailable`, `UnusedDependency`) — those still surface only in CI. Re-baselining (`qodana.sarif.json` updates) still happens by downloading the `qodana-report` workflow artifact from a CI run.
