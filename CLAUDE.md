# CLAUDE.md

This file provides guidance to Claude Code (claude.ai/code) when working with code in this repository.

## Status

Pre-1.0 Rust project (edition 2024). Vision: a game engine where Claude sits in a harness as assistant/engineer/designer. Architectural direction (see `docs/adr/`): a thin native **substrate** owns I/O, GPU, audio, and hosts a WASM runtime; engine **components** run as WASM modules and communicate via a **mail** system. (The whole system — substrate + components + tooling — is "Aether" or "the engine"; the substrate is just the native base layer.)

## Infrastructure crates (ADR-0069)

The non-component infrastructure cluster has four crates with role-distinct names:

- **`aether-data`** — universal data layer. Typed-id newtypes (`MailboxId`, `KindId`, `HandleId`), schema vocabulary (`SchemaType`, `KindShape`, `KindLabels`, `InputsRecord`, canonical bytes), `Kind` / `Schema` / `CastEligible` traits, `Ref<K>`, encode/decode helpers, native descriptor inventory. `no_std` + `alloc`. Every consumer that describes typed bytes depends on this.
- **`aether-codec`** — schema-driven JSON ↔ wire bytes (formerly `aether-params-codec`). Pure functions over `aether_data::SchemaType`; future home for non-JSON codecs (msgpack, save-format adapters).
- **`aether-hub-protocol`** — engine ↔ hub channel wire only (frames + framing helpers). Std-only.
- **`aether-kinds`** — substrate vocabulary: concrete kind types (`Tick`, `Key`, `DrawTriangle`, `aether.audio.*`, `aether.io.*`, `aether.control.*`, `aether.observation.*`, `aether.camera`) plus the well-known mailboxes registry (`mailboxes::CONTROL`, `mailboxes::DIAGNOSTICS`, `mailboxes::SINK_*`).

`aether-id`, `aether-mail`, and `aether-params-codec` are retired. `Mail<'_>` and `Sink<K>` live in `aether-component` (guest SDK); the host's dispatcher-side `Mail` lives in `aether-substrate-core`. The "transport envelope" abstraction is split between SDK and dispatcher rather than shared.

## Workflow

- **Exploration and design discussion** happens in chat with the user. No artifact required.
- **Planned work** (spikes, features, open investigations) lives in GitHub Issues. Referenced by the PR that closes them.
- **Load-bearing architectural decisions** are recorded as ADRs in `docs/adr/NNNN-title.md`. Use `docs/adr/TEMPLATE.md` when starting a new one. Number sequentially. An ADR is reviewed via a PR like any other change.
- **Branches**: `type/short-slug` (e.g. `chore/ci-bootstrap`, `feat/mail-runtime`, `docs/adr-workflow`).
- **Commits and PR titles** follow Conventional Commits (`type(scope): subject`). Enforced in CI against PR titles. Main uses squash-merge with PR title as the commit subject, so PR title quality matters.
- **Merging**: `main` is protected (PR required, all CI checks required, linear history, no force-push). Claude does not push to `main`, does not force-push reviewed branches, does not self-merge, and asks before destructive operations.
- **PRs** should be small and focused — one concept per PR.
- **Recursion in load-bearing code**: prefer iterative implementations (explicit work-stack/queue, arena-with-indices for tree data) over recursive ones in any algorithm whose depth could exceed a few hundred frames in practice. Recursion is OK for parse/AST walks where depth is structurally bounded by a small input file. Either way, recursive code on user-controlled or geometrically-derived data must enforce a depth/budget cap that returns an error rather than overflowing the stack.

## MCP harness

Claude drives a running engine through MCP — the concrete form of the "Claude-in-harness" vision. Starting `cargo run -p aether-substrate-hub` (and either letting the hub spawn substrates via `spawn_substrate`, or running `AETHER_HUB_URL=127.0.0.1:8889 cargo run -p aether-substrate-desktop` by hand — `aether-substrate-headless` for workloads that don't need a window or GPU, ADR-0035) exposes eleven tools to a Claude Code session pointed at the project-scoped `.mcp.json`:

- `mcp__aether-hub__list_engines` — connected engines (UUID + name/pid/version + `spawned` flag: `true` if the hub launched the process, `false` if it connected externally).
- `mcp__aether-hub__describe_kinds(engine_id)` — the kind vocabulary the engine declared at handshake, with enough structural detail to build params.
- `mcp__aether-hub__describe_component(engine_id, mailbox_id)` — receive-side capabilities a loaded component advertised (ADR-0033): typed handlers (id + name + per-handler doc extracted from the component's rustdoc `# Agent` sections), whether a `#[fallback]` catchall is present, plus the component's own top-level documentation. Read this before `send_mail` to pick a kind the component actually handles; strict receivers surface as `fallback: null`.
- `mcp__aether-hub__send_mail(mails)` — batched, best-effort. Each item carries `engine_id`, `recipient_name`, `kind_name`, and structured `params` — the hub looks up the kind's descriptor and writes bytes matching the wire format (`#[repr(C)]` for cast-shaped kinds, postcard for everything else). Response is a per-item status array; one failure doesn't abort siblings.
- `mcp__aether-hub__receive_mail(max?)` — non-blocking drain of observation mail the engine pushed to this session. Each item carries `engine_id`, `kind_name`, structured `params` (decoded against the engine's kind descriptor — symmetric to `send_mail`, ADR-0020), `payload_bytes` (always populated; primarily a fallback when `params` is null), an optional `decode_error` populated when decode failed, and a `broadcast` flag (`true` means fan-out to every attached session, `false` means targeted reply-to-sender). Read `params` first; fall back to `payload_bytes` only if it's null.
- `mcp__aether-hub__spawn_substrate(binary_path, components?, args?, env?, timeout_ms?)` — launches a substrate binary as a child of the hub with `AETHER_HUB_URL` injected. Blocks until `Hello` handshake; returns `engine_id` + `pid`. The hub owns the child for its lifetime. Optional `components: [{binary_path, name?}]` preloads are driven through the same path as `load_component`, sequentially in declared order; any load failure SIGTERMs the freshly-spawned child and surfaces a tool error so no half-loaded scene is ever returned. On success the reply includes a `components: [{mailbox_id, name, capabilities}]` array matching the input order.
- `mcp__aether-hub__load_component(engine_id, binary_path, name?, timeout_ms?)` — loads a WASM component by filesystem path. The hub reads the binary and forwards `aether.control.load_component` to the substrate, awaiting the `LoadResult` reply via the same pending-replies mechanism `capture_frame` uses. Returns `{mailbox_id, name}` inline on success. `binary_path` must exist as given (same rule as `spawn_substrate` — no `~`, no relative resolution). The component's kind vocabulary rides in the wasm's `aether.kinds` custom section (ADR-0028) — nothing to declare in the tool call. Rejects with "already in flight" if a prior load is pending on the session. Use this over `send_mail` — the path argument avoids inlining wasm bytes as MCP tool tokens.
- `mcp__aether-hub__replace_component(engine_id, mailbox_id, binary_path, drain_timeout_ms?, timeout_ms?)` — same shape as `load_component` but for ADR-0022 freeze-drain-swap. The hub reads the binary, forwards `aether.control.replace_component`, and awaits `ReplaceResult`. Kind vocabulary is embedded in the wasm (ADR-0028). `drain_timeout_ms` is accepted for wire compatibility but currently ignored — post-ADR-0038 the splice is structural (the old dispatcher's inbox sender is dropped, the dispatcher drains naturally, then joins), so there is no timeout knob to enforce. A future ADR may repurpose the field as a join-timeout if stuck guests become common.
- `mcp__aether-hub__terminate_substrate(engine_id, grace_ms?)` — SIGTERM → grace (default 2s) → SIGKILL. Errors on externally connected engines (the hub only terminates children it owns).
- `mcp__aether-hub__engine_logs(engine_id, max?, level?, since?)` — drain captured substrate `tracing` events for an engine (ADR-0023). Cursor-based polling: pass back the previous response's `next_since` to receive only new entries. `level` filters server-side (`"trace"|"debug"|"info"|"warn"|"error"`, default `"trace"`); `max` defaults to 100 / clamped to 1000. `truncated_before` flags hub-side ring eviction so a slow poller knows it missed a window. The buffer survives engine exit, so post-mortem polls work after a substrate crash. Substrate-side filter: `AETHER_LOG_FILTER` (standard `EnvFilter` syntax, e.g. `AETHER_LOG_FILTER=aether_substrate=debug,wgpu=warn`) overrides the INFO+ default.
- `mcp__aether-hub__capture_frame(engine_id, mails?, after_mails?, timeout_ms?)` — synchronous readback of the substrate's current frame as a PNG, returned inline as MCP image content. The substrate renders to an offscreen target and blits to the swapchain for presentation; the capture reads the offscreen, so it works regardless of window visibility (important on macOS where occluded windows would otherwise block). Optionally carries two mail bundles: `mails` dispatched *before* the capture (state-changing mail whose effects should appear in the frame) and `after_mails` dispatched *after* readback (cleanup such as restoring a flag the caller flipped for the capture). The substrate resolves every envelope in both bundles atomically — if any kind or recipient is unknown the whole request aborts with `Err` (no partial dispatch). The render loop's mail-queue-idle wait ensures the pre-capture bundle has been fully processed before the frame is read back, so the captured image reliably reflects its effects. Empty bundles mean "just capture" / "no cleanup". The hub diverts the substrate's `aether.control.capture_frame_result` reply off the general inbound queue via the await-reply mechanism. `timeout_ms` defaults to 5000, clamped to 30000. Rejects with "already in flight" if a capture is in progress on the same session; rejects with a clear error if the substrate has no GPU yet (request arrived before `resumed`). Use this to verify substrate rendering end-to-end without needing to look at the window, and to bundle "set X, show me, restore X" into one atomic tool call.

When verifying substrate behavior end-to-end, reach for the MCP harness before running a new test binary.

## Scenario runner (ADR-0067)

For Rust integration tests and CI gating that don't need a live MCP session, the **scenario runner** drives the same in-process substrate from a Rust thread. Four crates ship the surface:

- **`aether-substrate-test-bench`** — `TestBench::start()` boots a full chassis (scheduler, mail queue, wgpu offscreen render target) on the test thread, with a loopback channel attached to outbound so substrate replies route back without going through a hub. `advance(ticks)`, `capture()`, `send_mail<K>()`, `send_bytes(name, kind_id, bytes)`.
- **`aether-scenario`** — declarative `Script` of `Step`s (`Advance`, `Capture`, `Assert`, `LoadComponent`, `SendMail`) parsed from YAML. `Runner::run(&mut bench, &script)` walks the steps, encoding `SendMail` params via `aether-codec::encode_schema` against `aether_kinds::descriptors::all()` — same path the hub uses for `mcp__aether-hub__send_mail`, so any substrate kind is sendable from a script for free. Returns a `RunReport` with per-step pass/fail.
- **`aether-scenario-cli`** — `aether-scenario <path.yml>` boots a TestBench, runs the script, prints a per-step report, exits 0/1. The agent-facing entry point.
- **`aether-scenario-macros`** — `aether_scenario::scenario_dir!("scenarios")` proc-macro emits one `#[test]` per `.yml` file under the named directory. Component crates author scenarios as plain YAML and get IDE-runnable tests for free.

Scenarios need a wgpu adapter; CI installs `mesa-vulkan-drivers` on Linux runners and pre-builds component wasm before `cargo test`. Driverless dev boxes skip cleanly with an `eprintln!`. Reach for the scenario runner over a hand-rolled test binary when you want repeatable substrate verification in `cargo test`; reach for the MCP harness when you want exploratory or live observation.

Wasm components are discovered in CI as `cargo metadata` packages with `crate-type = cdylib` AND a dep on `aether-component` (issue 439). Both signals are structural — no filename convention to remember. Forgetting either signal means the crate isn't a component anyway.

The observation path (ADR-0008) goes the other way: engines emit to the well-known sink `"hub.claude.broadcast"` and the hub fans out to every attached session. The live substrate binary pushes `aether.observation.frame_stats` there every 120 frames — a good smoke test for `receive_mail`. Components reply to a specific sender (Claude session or other component) via the `reply_mail` host fn (ADR-0013, ADR-0017), wrapped by `Ctx::reply` in the SDK.

Input streams (tick, key, mouse_move, mouse_button) are publish/subscribe (ADR-0021): the substrate boots with empty subscriber sets and drops every input event until something subscribes. To wire a freshly-loaded component into the platform's event stream, mail `aether.control.subscribe_input` with `{ "stream": "Tick" | "Key" | "MouseMove" | "MouseButton", "mailbox": <id from load_result> }`. Multiple components may subscribe to the same stream; the substrate fans out to every subscriber. `aether.control.unsubscribe_input` removes one. Both reply via `aether.control.subscribe_input_result`. Subscriptions are auto-cleared when a component is dropped and preserved across `replace_component` (the mailbox id is stable).

`aether.control.replace_component` is a structural splice (ADR-0022 + ADR-0038): the substrate splices the mailbox's inbox over to a new dispatcher, drops the old sender, and joins the old dispatcher once it drains naturally. Mail that arrives during the splice lands in the new dispatcher's inbox. The `drain_timeout_ms` field is accepted on the wire but currently ignored — post-ADR-0038 there is no freeze-drain timeout to enforce. A future ADR may repurpose it as a join-timeout if stuck guests (e.g. a wasm `deliver` in a CPU-only loop) become common.

Window + platform state is mailed, not MCP-toolled. `aether.control.platform_info` returns a `PlatformInfoResult` snapshot (OS + arch, engine version + kind count, GPU adapter/limits, full monitor list with available video modes, current window mode + size). (The wire also carries an `EngineInfo.workers` field; post-ADR-0038 there is no shared worker pool — one OS thread per component — so the value is retained for wire stability but no longer reflects scheduler parallelism.) `aether.control.set_window_mode` switches the window at runtime between `Windowed { width?, height? }`, `FullscreenBorderless`, and `FullscreenExclusive { width, height, refresh_mhz }`; the substrate resolves exclusive modes against the window's current monitor and replies with the actually-applied size. `aether.control.set_window_title` updates the window title; the reply echoes the applied title back. Boot-time overrides: set `AETHER_WINDOW_MODE` to `windowed`, `windowed:WxH`, `fullscreen-borderless`, or `exclusive:WxH@HZ` (integer Hz) to pick the initial mode before any component loads, and `AETHER_WINDOW_TITLE` to set the initial title (default `aether`). `platform_info`, `set_window_mode`, `set_window_title`, and `capture_frame` are desktop-chassis-only; the headless chassis replies `Err { error: "unsupported on headless chassis…" }` (ADR-0035) so callers fail fast instead of hanging on a reply that never comes.

The headless chassis (`aether-substrate-headless`, ADR-0035 Phase 3) drives ticks from a std timer instead of winit vsync; cadence defaults to 60 Hz and is overridable via `AETHER_TICK_HZ`. It inherits the same hub client, mail scheduler, and `aether.control.{load,drop,replace}_component` surface as desktop, so components (and the dev workflow of loading-via-MCP) port across chassis unchanged. A nop `render` sink is pre-registered so desktop-designed components that emit `DrawTriangle` every tick don't warn-storm `engine_logs`; the `camera` sink gets the same nop treatment for the same reason.

Vertex geometry is world-space. The substrate holds a 4×4 `view_proj` uniform (column-major, 64 bytes) that the vertex shader applies to every vertex: `clip_pos = camera.view_proj * vec4(position, 1.0)`. The uniform defaults to identity, so components that still emit clip-space-ish world coords render unchanged until a camera component starts publishing. Camera components send `aether.camera { view_proj: [f32; 16] }` to the desktop chassis's `aether.sink.camera` sink — latest value wins, uploaded to the GPU uniform before each draw. Fire-and-forget: no reply. `Vertex` grew a `z: f32` field (24 bytes on the wire; `DrawTriangle` is 72 bytes per triangle). The reference camera implementation is `aether-camera-component`, a multi-camera/multi-mode component: it hosts N named cameras (each in `Orbit` or `Topdown` mode) and publishes only the active one's `view_proj` per tick. Boots with one default camera (`name: "main"`, orbit) marked active, so loading the component immediately yields a visible camera (~12 s/revolution auto-orbit, matching the prior single-mode behaviour). Driver surface: `aether.camera.create { name, mode: ModeInit }`, `aether.camera.destroy { name }`, `aether.camera.set_active { name }`, `aether.camera.set_mode { name, mode }`, and per-mode delta kinds `aether.camera.orbit.set { name, params: OrbitParams }` and `aether.camera.topdown.set { name, params: TopdownParams }`. `OrbitParams` / `TopdownParams` use `Option<...>` per field, so the same struct seeds creation (compiled defaults fill unspecified fields) and streams partial deltas — `OrbitParams { distance: Some(5.0), ..None }` zooms without disturbing yaw / pitch / target. Inactive cameras still tick (orbit yaw keeps accumulating) so re-activating doesn't snap. Mode mismatch on `*.set` warns and no-ops; switch with `set_mode` first. Per ADR-0066 the kind types and parameter structs live in the `aether-camera` trunk rlib (the runtime cdylib stays in `aether-camera-component`); other components depend on `aether-camera` for the wire shape, never on the cdylib.

Vector and matrix math lives in the `aether-math` crate: `Vec2`, `Vec3`, `Vec4`, `Mat4`, `Quat`, `Aabb` (column-major, YXZ Euler, right-handed Y-up, wgpu `[0, 1]` clip-space depth, scalar `f32`, `no_std`). Reach for it before writing inline `cross` / `dot` / `normalize` / `rotate` / aabb checks. When something's missing, add it to `aether-math` rather than reinventing locally — domain-agnostic spatial/numeric primitives only (mesh, render, CSG, audio types stay in their owning crate). `[f32; 3]` survives at wire boundaries (`Vertex`, AST nodes, the OBJ exporter) and bridges via `Vec3::from_array` / `to_array`; everything in between should use the typed forms.

Depth testing is on in the desktop chassis: a `Depth32Float` texture paired with the offscreen color target, cleared to 1.0 each frame, with `LessEqual` comparison. Combined with the topdown camera (which looks down `-Z`) this means larger world-z draws on top of smaller world-z — the intuitive "push things up to bring them forward" convention. Floors / backdrop geometry sit at `z = 0`; movers / foreground should sit at `z = 0.1` or higher. The reference `aether-player-component` draws its body at `z = 0.1`.

Mesh authoring goes through the mesh viewer component (`aether-mesh-viewer-component`, with `aether.mesh.load` and any future driver kinds living in the `aether-mesh-viewer` trunk rlib per ADR-0066), a thin file-loader for both the s-expression mesh DSL (ADR-0026 + ADR-0051) and Wavefront OBJ. Load the wasm with `load_component` and pick whatever name fits the role — post-ADR-0058 the bare names `render` / `camera` / `audio` / `io` / `net` / `handle` are no longer reserved (chassis sinks moved to the `aether.sink.*` namespace). Drive the loaded viewer with one mail kind: `aether.mesh.load { namespace, path }` asks the viewer to fetch bytes through the `aether.sink.io` sink (ADR-0041 namespaces) and parse them on reply, dispatching on extension — `.dsl` runs through `aether-mesh`'s parser+mesher and emits polygon-edge wireframes alongside filled triangles, `.obj` runs through a built-in fan-triangulation parser with a default soft-blue color and no wireframe. The DSL agent loop is now write-file-then-load: emit the DSL via `aether.io.write { namespace, path, bytes }`, then `aether.mesh.load` against the same path. Fire-and-forget. Each successful load parses, meshes, and atomically replaces the cached triangle list; partial parse / mesh failure keeps the previous mesh visible and surfaces the error via `engine_logs`. Every tick the viewer replays the cached triangles to the `aether.sink.render` sink, so once a mesh is loaded it stays on-screen with no further mail. Colors are DSL palette indices (`:color N`, `u32`) mapped via an 8-entry built-in palette wrapped modulo length. Vocabulary covers cube / cylinder / cone / sphere / wedge primitives, `extrude` for face extrusion, `mirror` / `array` for repetition, and `sweep` for path-driven prism generation (parallel-transport framing is normative — ADR-0051). Examples live in `crates/aether-mesh/examples/` (box, lamp_post, teapot). The crate `aether-mesh` is library-only (parser + mesher); the viewer component is the substrate-side wrapper. Boolean composition (`union` / `intersection` / `difference`) was retired by ADR-0062 and lives on the `archive/csg-bsp` branch.

File I/O is a mail-based sink (ADR-0041). The substrate owns an `aether.sink.io` sink and an `AdapterRegistry` mapping logical namespaces to storage backends. v1 ships three namespaces — `save://` (writable, per-user persistent — default `dirs::data_dir()/aether/save`), `assets://` (read-only — default `{current_exe}/../assets`), `config://` (writable — default `dirs::config_dir()/aether`) — each with a `LocalFileAdapter` that normalizes paths (`..` / absolute prefixes → `Forbidden`) and writes atomically via tmp+rename. Components mail `aether.io.read { namespace, path }` / `write { namespace, path, bytes }` / `delete { namespace, path }` / `list { namespace, prefix }` to the `aether.sink.io` mailbox (the kind namespace `aether.io.*` is *not* the mailbox name — it's the kind id namespace; the mailbox is the `aether.sink.*` form per ADR-0058). Each reply kind echoes the originating `namespace` + `path` (or `prefix` for `List`) on both `Ok` and `Err` variants so components correlate reply-to-request by the echoed fields + the reply kind itself — no pending-op queue, no sink-dispatch-order FIFO dependency. `WriteResult` omits `bytes` from the echo so a megabyte write still produces a small reply. On malformed-request decode failure the echo collapses to empty strings and `IoError::AdapterError` carries the diagnostic. `IoError` is one of `NotFound` / `Forbidden` / `UnknownNamespace` / `AdapterError(String)`. Namespace in the mail is the short prefix (`"save"`), not `"save://"` — the double-colon form is a UX convention in docs and logs. Desktop and headless chassis both wire the sink; hub chassis doesn't (mail to `aether.sink.io` warn-drops as unknown mailbox). Env-var config: `AETHER_SAVE_DIR`, `AETHER_ASSETS_DIR`, `AETHER_CONFIG_DIR` override the defaults. Adapter calls run synchronously on the sink dispatch thread — fine for KB-MB save/config files; asset streaming would want a host-fn fast path, deferred. TOML + CLI layers in the precedence stack (CLI > env > TOML > defaults) are committed but not shipped in v1.

Audio is a desktop-chassis peripheral (ADR-0039). The substrate owns a `cpal` output stream fed by a hand-rolled synth: one voice per `(sender_mailbox, instrument_id, pitch)` triple, ADSR envelopes, flat mix scaled by master gain, soft-clipped via `tanh`. v1 ships five built-in instruments (`sine_lead`, `square_bass`, `triangle`, `saw_lead`, `pluck`) indexed by `NoteOn.instrument_id`; runtime-defined patches are parked. Components emit `aether.audio.note_on { pitch, velocity, instrument_id }` and `aether.audio.note_off { pitch, instrument_id }` to the `aether.sink.audio` mailbox — fire-and-forget cast kinds; the voice-key keeps notes from different senders independent. `aether.audio.set_master_gain { gain }` also flows through the `aether.sink.audio` mailbox and replies with `SetMasterGainResult::Ok { applied_gain }` on desktop or `Err { error }` on headless / hub / audio-disabled substrates. Mixing and effects are user-space composition — the substrate sums voices flat, so a music component that wants ducking or buses forwards events through its own mail surface. Env vars: `AETHER_AUDIO_DISABLE=1` skips cpal init (nop sink, `SetMasterGain` replies Err); `AETHER_AUDIO_SAMPLE_RATE=<hz>` requests a specific rate (default: device preference, typically 44100 or 48000).

**Recipient-name convention.** Chassis-owned sinks live under the `aether.sink.*` namespace (ADR-0058) — `"aether.sink.render"`, `"aether.sink.camera"`, `"aether.sink.audio"`, `"aether.sink.io"`, `"aether.sink.net"`, `"aether.sink.handle"`. The bare names (`"render"`, `"camera"`, etc.) belong to user-space components: load a camera component as `"camera"` and it does not collide. Inside the hub's MCP `send_mail`, `recipient_name` names the mailbox; `kind_name` names the payload shape. Note that mailbox name and kind namespace are deliberately *different*: send `aether.audio.note_on` (kind) to `aether.sink.audio` (mailbox), or `aether.draw_triangle` (kind) to `aether.sink.render` (mailbox) — the `sink` infix is what disambiguates the two in mail logs. Control-plane kinds (ADR-0010 / ADR-0039 reply kinds) still go to `"aether.control"` — its name predates ADR-0058 and was already namespaced.

config precedence is CLI > env > TOML > defaults with v1 shipping env + defaults only), ADR-0026 (DSL-only mesh representation — Lisp s-expression primitive composition is the authoring path; SDF/voxel/CSG rejected for v1), ADR-0051 (DSL mesh primitives + sweep — full v1 vocabulary including parallel-transport framing for sweep), and ADR-0052 (mesh editor is the DSL — DSL text + hot reload supersede the stateful vertex/face graph; the editor component was later folded into `aether-mesh-viewer-component` which loads `.dsl` and `.obj` files via `aether.mesh.load`).

The component FFI surface uses a `_p32` suffix on every pointer-typed import (`aether::send_mail_p32`, `reply_mail_p32`, `save_state_p32`) and on the `receive_p32` / `on_rehydrate_p32` exports. Non-pointer exports (`init`, `on_replace`, `on_drop`) are unsuffixed. The suffix locks the wasm32/wasm64 naming convention without committing to dual registration today — see ADR-0024 for the deferred Phase 2. `send_mail_p32`'s `recipient` and `kind` are scalar `i64` / `u64` — ADR-0029 made mailbox ids 64-bit name hashes and ADR-0030 made kind ids 64-bit schema hashes; neither falls under the `_p32` (pointer-width) convention. `resolve_mailbox_p32` was retired in ADR-0029; `resolve_kind_p32` was retired in ADR-0030 Phase 2. Guests get `Kind::ID` and `mailbox_id_from_name` as compile-time consts — no host-fn round trips for address resolution. `init` gained a `mailbox_id: u64` parameter so the SDK's init walker can self-address the `aether.control.subscribe_input` mail it emits for every `K::IS_INPUT` kind (ADR-0030 Phase 2).

Components declare their receive-side kind vocabulary with `#[handlers]` on a single `impl Component for C` block (ADR-0033). Each handled kind is a `#[handler] fn on_whatever(&mut self, ctx: &mut Ctx<'_>, mail: K)` method — the decoded payload arrives as the third parameter, so the kind is inferred from the signature and the author never writes a typelist or an `is::<K>()` branch. An optional `#[fallback] fn on_anything(&mut self, ctx, mail: Mail<'_>)` catches everything else; omit it for a strict receiver. The macro codegens a dispatch table and emits an `aether.kinds.inputs` wasm custom section enumerating every handler (id + name + per-handler rustdoc extracted from `# Agent` sections) — that's what `describe_component` surfaces over MCP, so Claude can ask "what does this component listen for?" without running it. `Sink<K>` keeps its explicit field for sends (mailbox names are data, not types). ADR-0027's `type Kinds` typelist and the generic `Component::receive` dispatch method are retired — handlers are the sole source of truth.

## Commands

- Build: `cargo build` (release: `cargo build --release`)
- Run: `cargo run -p <crate>` — workspace root has no default binary; use `-p aether-substrate-hub`, `-p aether-substrate-desktop`, or `-p aether-substrate-headless`.
- Test: `cargo test` (single test: `cargo test <name>`; single-threaded with output: `cargo test -- --nocapture --test-threads=1`)
- Lint: `cargo clippy --all-targets -- -D warnings`
- Format: `cargo fmt` (check-only: `cargo fmt -- --check`)
- Type/borrow check only: `cargo check`
